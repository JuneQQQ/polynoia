Conflict Resolution · Design Internals

多 Agent 冲突解决方案 · 底层设计原理

多个 AI Agent 像同事一样在同一个工作区并行干活,冲突无可避免——而且不止一种。 Polynoia 把冲突拆成四个正交的层,在四个不同的时间点拦截: 意图 → 文件 → 代码 → 语义,= 分派 → 隔离 → 合并 → 验收。 每一层都可独立失败而不击穿后续层。

中心哲学 —— 只为真实的争用付费。 两个 agent 改不同文件、目标不打架 → 零协调开销; 只有当他们的意图、同一文件、同一行代码或同一处语义真正撞上, 系统才掏出对应那一层的协调成本。没有全局锁,没有把所有写串行化—— 代价精确落在真正冲突的那个点上。底座是 git:它是唯一事实源,不是自研的冲突引擎。
意图 · orchestrator 拆解 文件 · 分支隔离 + 写闸 代码 · git 真合并探测 语义 · LLM 修复 + 验收
Defense in Depth · Timeline

一次并行协作里,四类冲突在四个时间点被拦截

从用户发话到产物落地,任务流经四道闸。每道闸只管自己那类冲突,放行的交给下一道。

Layer 1 · Intent

① 意图冲突 —— 在"动手之前"对齐目标

两个 agent 若对"该做什么"理解不同,后面所有的隔离与合并都是白费。所以第一道闸在 任务分派期:Orchestrator(本身就是一个 role=orchestrator 的 Agent) 把用户意图拆成互不冲突的子任务,并用共享契约锁定边界。

①
分派期

机制 · 拆解 / 讨论 / 表决

  • dispatch —— Orchestrator 并行派活,每个子任务带 contract(锁定的边界条款), workers 在各自分支并行展开,不互相踩。
  • discuss —— 需要先对齐时开圆桌,多 agent 往返讨论收敛意图,再动手。
  • ask_user —— 意图本身有歧义 → 阻塞并向用户发 ask-form,人来拍板。
锚点
· mcp/tools.py · _DispatchTool / _DiscussTool / _AskUserTool
· api/routes.py · dispatch drain → BurstCard + spawn workers
· contract 注入每个 worker 的 prompt(共享边界,防意图漂移)

拦截时间点:任务分发时 —— 把潜在的意图冲突消解成"不重叠的子任务",从源头减少后续撞车。

Layer 2 · File

② 文件冲突 —— 在"落盘之前"隔离写入

多个 agent 想改同一文件?第二道闸在写入期:每个 agent 在共享工作区里有自己的 git worktree + 分支,并行写互不可见;manual 模式下每次写还要先过用户审批闸。

②
写入期

机制 · 分支隔离 + 写闸 + 文件锁

  • per-agent worktree 分支 agent/<id>/conv-<conv>:从 main 切出, 各 agent 的写落在自己分支,撞车被推迟到"集成时"而非"编辑时"。
  • pending-edit 闸(manual 模式):write 落盘前 POST 一条待审编辑、 长轮询等用户 ✓/✗;auto 模式直接放行。
  • per-(conv,file) 锁:同进程内对同一文件的写串行化,防写撕裂。
锚点
· sandbox/_core.py · create_workspace_sandbox(worktree + 分支)
· mcp/tools.py · _gate_via_pending_edit / ToolContext.file_lock
· api/routes.py · /api/pending-edits(create / wait / decide)

拦截时间点:落盘前 —— 隔离让大多数"同时改"根本不构成冲突;真撞上的留给第三层。

Layer 3 · Code (git merge)

③ 代码冲突 —— 合并时用真 git 探测,不留痕

分支隔离把撞车推迟到合并那一刻。第三道闸在合并期:对每条分支跑一次真 3-way 合并, 干净则静默落地,冲突则抓三方 blob 冻成一等数据、再把现场抹平。没有自研行 diff——判定权交给 git。

# probe_merge(branch) —— sandbox/_core.py
git -c merge.conflictStyle=diff3 merge --no-commit --no-ff <branch>
git diff --name-only --diff-filter=U          # 有 U(unmerged) 文件? = 冲突
git show :1:<path>  # base   |  git show :2:<path>  # ours(main)  |  git show :3:<path>  # theirs(分支)
git merge --abort                              # ★ 抹平现场,main 一字未动
✅

clean

无 U → 直接 commit,分支静默落地 main。

⚠️

conflict

有 U → 抓三方 blob → abort → 冻成 ConflictRow。

🔒

串行锁

合并按 workspace_id 串行(共享 .git 只有一个 HEAD)。

🧯

不留痕

每条路径结束 main 与探测前完全一致;崩溃恢复先验 MERGE_HEAD 再 abort。

四类 ctype · 由三方 blob 的存在性推导

ctype判定携带数据谁能解
contentbase/ours/theirs 三方都在markers(diff3)+ 三方全文LLM 语义合并
add_add无 base(两侧都新建)ours + theirsLLM 合并/选边
modify_delete缺一侧存活侧 + baseLLM 判断/留人
binary含 NUL,不可 decode无文本永远人工
锚点 · sandbox/_core.py · probe_merge / conclude_merge / _capture_conflict_file · api/routes.py · _drain_unmerged_branches / _surface_conflict · workspace_merge_lock

拦截时间点:分支合并时 —— 文本层撞车冻成可见、可裁决、可回滚的冲突卡。

Layer 4 · Semantic

④ 语义冲突 —— 文本能合,但逻辑对不上

最难的一类:git 合得干干净净,可两段改动放在一起行为是错的(接口不匹配、违反契约、双方各加了一半校验)。 这层靠 LLM 的判断 + 验收闭环:既在冲突里做语义合并,也在合并后做核对。

④
合并 / 验收期

机制 · LLM 修复 + report + critic + present

  • LLM 自动修复轮 NEW —— content/add_add 冲突不是机械选边,而是让分支作者 读懂两侧意图、产出语义正确的合并文本(见下一节)。
  • report 自评 —— 每个 worker 用 report 上报 status/deliverables/contract_ok; 没报告 = "未验证"对待,不隐瞒。
  • critic / auditor 角色 —— 只读 + report 的审查 tier(_TIER_AUDITOR),专做核对不做改。
  • orchestrator 验收 + present —— 汇总轮核对每条 report 的 contract_ok、抽查产物,再统一 present 已验收的成果。
锚点
· api/routes.py · _build_conflict_fix_prompt · mcp/tools.py · resolve_conflict(LLM 语义合并)
· mcp/tools.py · _ReportTool / _RecallTool / _TIER_AUDITOR / _PresentTool
· api/routes.py · is_last → 验收 summary → _maybe_handoff_to_orchestrator

拦截时间点:合并时(语义合并) + 合并后(验收) —— 文本通过不等于逻辑通过。

Deep Dive · Auto-Fix Round

聚焦:LLM 自动修复轮 —— 让 agent 自动做人会做的事 NEW

在 merge_mode='auto' 的对话里,代码冲突 surface 后不直接停下等人—— 系统 fire-and-forget 拉起分支作者修一轮;修好则卡翻绿静默落地,修不好/二进制才留给人。 它一字不改地复用人工 resolve 的全路径。

三条非显然的不变量(本设计的关键)

🔁

① 为何必须把三方数据内联进 prompt

作者的 worktree 是 merge --abort 之后的分支侧,磁盘上根本没有冲突标记, 也看不到 main(theirs)侧。read 工具无法做三方合并,冲突三方必须随 prompt 送达 (_build_conflict_fix_prompt,带 size cap;全二进制/过大则回退留人)。

⚡

② 为何 spawn 必须 fire-and-forget

drain 此刻正持有 workspace_merge_lock。若 await 修复 turn, 它的 resolve_conflict → /resolve 会再抢同一把锁——而 asyncio.Lock 不可重入 → 自死锁。用 _spawn_turn(create_task)让 drain 先释放锁,修复 turn 跨 task 排队,无环。

🛡️

③ 为何不会无限循环

drain 会跳过"已有 open/resolving/abandoned 卡"的分支(already guard)。 每个冲突至多触发一次自动修复;失败回 open 后仍在 already 集合 → 后续 drain 不重试, 优雅降级为"留人手动解"。修复 turn 不挂 burst 参数,不污染 is_last 状态机。

解决到底是什么操作?—— 三策略,不是「覆盖」也不是「git 死合」

很多人以为冲突解决=选一边覆盖。实际上 conclude_merge(branch, resolutions, sides, deletions) 提供 per-file 三选一,决策方(auto=LLM / manual=人)按文件挑;覆盖只是其一,语义合并才是主路。

策略入参是什么用在
语义合并(主力)resolutions = {path: 完整文本}读三方,产出合并后的完整文件(如 union 两边的 key)—— 非覆盖、非死板拼接content / add_add
选边覆盖sides = {path: 'ours'|'theirs'}整侧取一(checkout --ours/--theirs)—— 这才是「直接覆盖」二进制 / 一方明显赢
删除deletions = [path]接受删文件(git rm)modify/delete

可混用(一次 resolve 里不同文件用不同策略)。落地:在当前 main 上重新 git merge --no-commit 再 apply 选择 + marker 残留校验 → 生成双父 merge commit;binary 永远人工,modify_delete 允许 LLM 放弃 → 留 open。

Commit History · Provenance & Tree

提交历史:谁在哪条分支做了什么、怎么合进 main

冲突闭环的产物最终是 git 提交。提交历史面板不只是 diff 浏览器 —— 它还要如实回答 归属(谁做的)、位置(在自己分支还是 main)、结构(分支怎么分叉、怎么合回)。 底座仍是 git:workspace_commits 读真实拓扑,不另存账本。

👤

① 归属:按 persona 署名

agent 写文件的提交改用 persona(turn_agent_id)署名,不再是共享适配器 (claudeCode/codex)→ 历史显示 顾屿/沈昭/周野。冲突解决提交由 conclude_merge(author=resolved_by) 归属到解决它的 agent,不再是泛泛的「你」。

每条提交带 lane:分支 = agent 自己 worktree 上的内容提交;main = 合并 / 解决 / 初始化 / 用户编辑(落在 main)。 一眼看出哪些是分支私活、哪些是集成。

Rollback · Defense in Reverse

四道撤销:从轻到重

⤺

① 探测守卫 abort

探测撞冲突立即 merge --abort,main 从未改变。自动发生。

⊘

② pending-edit 拒绝

manual 模式点 ✗,改动根本不 commit——落盘前拦下。

✕

③ abandon 冲突

/conflicts/{id}/abandon → abandoned,分支保留不合并。

⟲

④ restore main

restore_main_to 做 reset --hard + 写 refs/polynoia/undo/<ts> 安全网。最重。

↩︎

单卡 diff「撤销」= patch 反向 apply,合并后会优雅降级

diff 卡的「撤销」是 git apply --reverse 反打这张卡的补丁。一旦该文件在这笔之后又被改过 (尤其经过冲突 union 合并),补丁上下文对不上、反 apply 必然失败 —— 这是 git 本性,不是 bug。前端因此不再抛生硬报错, 而是隐藏注定失败的按钮、提示「该笔已被后续改动覆盖」。合并后正确的撤销是 commit 级:消息上的 回到这个对话(= 第④道 restore_main_to)。

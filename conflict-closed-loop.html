The Gap

修补之前:一条「静默的死胡同」

在群聊 + 工作区模式下,每个 (agent, conv) 在自己的分支 agent/{id}/conv-{id} + worktree 上干活。一个 burst(并行任务批次)的所有 worker 完成后要逐分支合并进 main。旧路径是这样的——

BEFORE · 旧路径

合并撞冲突 → 悄悄丢弃

  • 逐分支 git merge --no-ff
  • 任何非零 rc → git merge --abort 丢弃改动
  • 标一个 needs-manual 然后……无人接手
  • 整段裹在 try/except 里,异常被吞 —— 用户什么都看不到

代价:rule.md 直接考核「代码冲突处理」;一个 agent 的工作就这么人间蒸发了。

AFTER · 冲突闭环

冻成一等数据 → 可见可解可回滚

  • 真 git 探测冲突(不留半合并)
  • 冻成一等 conflict 卡 + ConflictRow(单一事实源)
  • 在 IM 流里对所有人可见
  • LLM 自动修 或 用户在 diff 视图手动解 → 再真合并一次
  • 同卡翻 resolved + 系统卡确认

挂载点:api/routes.py:_merge_burst_to_main —— 唯一真实合并点。

Four Kinds of Conflict

四类冲突,四套打法

不是所有「冲突」都是一回事。Polynoia 把它们分成四层,越往下越难、越往前置。关键认识:结构冲突是已解决问题(别再发明检测器),语义/意图冲突无法被机器完备捕获(诚实承认),所以它们被推到「动手之前」和「测试 + 人裁决」。

① 文件冲突 / 写争用

两个 agent 同时写同一个文件。

怎么解
  • 每 (conv, file) 一把 asyncio.Lock —— 单进程内串行同一文件的写
  • write 是唯一被审计的写路径(search-replace edit 工具已退役)
  • 更深层的并发靠下面的「分支隔离」消解

mcp/tools.py · ToolContext._file_locks

🌿
确定性 · 已落地

② 代码冲突 / 结构冲突

两条分支改了同一段代码,合并撞车。

怎么解
  • 用 git 自己的合并引擎做探测,不自建正则/AST 探测器
  • probe_merge → 真 3-way 合并 → 冲突冻成一等数据
  • 零漏报零误报(铁律一:结构冲突=已解决问题)

sandbox/_core.py · probe_merge / conclude_merge

🧠
不可完备检测 · 部分 / P2+

③ 语义冲突

文本合并成功,但逻辑坏了。
(A 删了函数 / B 新增了对它的调用)

怎么解
  • 诚实承认漏检:write skew 是定理级漏检,写集比较必然漏
  • 唯一务实解 = Bayou 式重跑受影响测试(可执行断言)
  • P1/P2 只保证文本合并成功;语义验证推 P2+

铁律二:语义冲突 = 定理级漏检

🎯
前置消解 · 已落地

④ 意图冲突

两个 agent 对「该做什么」理解不一致
(各自发明字段名 / 端口 / API 形状)。

怎么解
  • 不在合并期解决 —— 在动手之前就锁定共享规格
  • Orchestrator 派发时带 contract,逐字交给每个队友
  • 共享记忆 conv_memory 让全队读同一份 decision/artifact

ADR-014/015 · dispatch contract + conv_memory

💡 为什么这样分? CALM 定理:协调只在「非单调」操作上才需要。纯新增(新文件 / 新符号 / append)是单调、可交换、免冲突直接合;只有改 / 删 / 重命名共享符号才撞冲突。所以 burst 产物大多走 clean 路径,重武器只花在真冲突的那一小撮分支上 —— 协调成本只按真实写争用付费。
Defense in Depth

三层防御:从「不撞」到「撞了能解」

同一份代码被多 agent 并发改,防御不是单点,而是三道由浅入深的关卡。前两道尽量让冲突不发生,第三道把真发生的冲突变成可解的数据。

1

进程内文件锁 — 同一文件的写,串行化

每个解析后的文件路径维护一把 asyncio.Lock。同一对话里两个 agent 想写同一个文件?排队,一个一个来。这道锁挡住的是「同一进程内的同文件写撕裂」。

ToolContext.file_lock(path) · _file_locks: dict[str, asyncio.Lock]

2

per-agent 分支 + worktree — 物理隔离,各改各的

每个 (agent, conv) 拿到独立分支 agent/{id}/conv-{id} 和独立 worktree。多 agent 真并发,谁也踩不到谁。新文件 / 新符号天然单调 —— 合并时直接 clean。这是 ADR-003「workspace 共享 git」的地基。

sandbox/_core.py · git worktree add -b agent/{id}/conv-{id}

3

真 git 3-way 合并探测 — 撞了,变成一等冲突数据

前两层挡不住的「同段代码两路改」,在合并时由 git 合并引擎本身探测。冲突不再是阻塞态或被丢弃的异常 —— 它被冻成 ConflictRow,进入下面的闭环。这就是 冲突闭环 的核心。

probe_merge → ConflictRow → conflict card → conclude_merge

✅ 注意一个刻意的「不做」: 服务端永远不写 3-way merge 算法。自己写合并算法容易引入静默 bug;Polynoia 只调用 git 的合并引擎(确定性、零误报),解不开的交给 LLM 或人。简单可靠 > 自作聪明。
The Closed Loop

冲突闭环五阶段:探测 → 冻结 → 可见 → 解决 → 再合并

这是整个设计的主干。注意贯穿始终的那把橙色锁 —— 整个 workspace 跨所有 worktree、所有对话只有一个 HEAD / index,合并只能在 root 串行进行。锁的键是 workspace_id 而不是 conv_id。

永不停在半合并态。这是整个设计的正确性命门。 诚实边界(底栏) 结构冲突 确定性 · git 引擎 · 零误报 → 本设计核心,已落地 语义冲突 定理级漏检 · 靠测试探 → P2+,有覆盖上限 意图冲突 共享契约 / 账本 · 动手之前 → ADR-014/015,已落地
状态机:四态,一张卡(稳定 message_id)从头跟到尾

探测的关键 git 机制(git 2.25.1 实测验证)
# probe_merge(branch) — 探测但绝不留痕,root 探完仍是干净可合并态
git checkout main
git -c merge.conflictStyle=diff3 merge --no-commit --no-ff <branch>
  ├ rc==0 且无 U 文件 → CLEAN:git commit --no-edit               # 同今天 success
  └ rc!=0 → CONFLICT:逐个冲突文件抓三方 blob
       markers = 读 worktree 文件(含 <<<<<<< ||||||| ======= >>>>>>>)
       base   = git show :1:<path>   # add/add 无基 → rc=128 → base=None,绝不把 stderr 当内容
       ours   = git show :2:<path>   # main 侧
       theirs = git show :3:<path>   # branch 侧
       git merge --abort             # ★ 实测:root 回到 status 空 + 无 MERGE_HEAD(干净!)
Not Just Text

五类 git 冲突 —— 文本合并模型对多数类型无效

实测发现:git 的冲突不止「文本内容冲突」一种,各类行为迥异。盲目套「让 LLM 删 <<<<< 标记」对 modify_delete / binary 只会产出垃圾。所以每个冲突文件先分类,再分派到对应的解决 UI。

类型 ctype检测特征(实测)blob 状态解决 UI
内容 content worktree 有 <<<<<<< 标记 base/ours/theirs 齐(或 base=None) @git-diff-view Split + 编辑去标记
双新增 add_add 双方都新增同名文件,无基 git show :1: rc=128 → base=None ours / theirs 二选一 或 编辑
改/删 modify_delete status DU/UD,worktree 无标记 一侧 blob 缺失(tombstone) keep-file / delete-file 单选(文本合并无意义)
重命名 rename rename/rename、rename/delete 路径不一,blob 错位 选保留哪个路径/内容
二进制 binary Cannot merge binary files,无标记 二进制,不可 UTF-8 decode 仅 take-ours / take-theirs;blob 存 base64
⚠️ 必修坑: _workspace_run 的 .decode('utf-8','replace') 会损坏二进制。binary 检测靠前 8000 字节找 NUL 字节 —— 命中就不 decode、限 take-side、blob 走 base64。modify_delete 强行走「删标记」prompt 只会生成垃圾,所以必须走 keep/delete 选择分支。
How Agents Are Managed

协作怎么管:burst 并行车道 + 谁最后谁触发合并

Orchestrator 是一个 role="orchestrator" 的 Agent(不是特殊代码),它把任务拆成子任务、并行派发。一次并行派发 = 一个 burst,每个 worker 在自己的车道上跑,互不交织。

burst 注册表 = 协作的状态机

每个并行批次在 _conv_bursts 注册一条记录:

_conv_bursts[conv_id][tp_id] = {
  "payload":    TasksPayload,   # 任务数组
  "pending":    {task_id, ...}, # 还没做完的
  "orch":       orch_agent_id,
  "workspace_id": ws_id,
  "contract":   shared_spec,   # 共享契约
}

每个 worker 完成 → _mark_burst_task(..., "done") 把自己从 pending 里划掉。

🔑 is_last 的竞争安全: is_last = not reg["pending"] 必须同步判定 + 立即 pop(在 await 之前!),否则两个 worker 同时看到 pending 空,都以为自己最后 → 双触发合并踩坏 .git。

两种合并模式 —— 只决定「谁先解」,不改数据路径

🤖 auto 模式(默认)

  • MCP 写入直接落分支 + commit
  • burst 完成 → 自动 probe_merge
  • 撞冲突 → 先让 LLM 修复轮试一次(P2)
  • LLM 失败 → 回落人工,同一行接管

✋ manual 模式

  • 每个写入经 _gate_via_pending_edit 闸门
  • 挂 HTTP 长轮询,等用户 ✓/✗
  • 推 PendingEdit 卡给用户审批
  • 合并仍走同一条 probe → conflict 路径

Conversation.merge_mode(默认 auto,可继承 Workspace.default_merge_mode)· PATCH /api/conversations/{id}/merge_mode

Pre-Action, Not Merge-Time

意图冲突:在动手之前就对齐,而不是在合并时收拾

最贵的冲突不是「两段代码撞了」,是「两个 agent 对该做什么理解不一致」—— 一个把金额当浮点、一个当整数;一个叫 userId、一个叫 uid。这种意图冲突合并期才发现就太晚了。Polynoia 把它前移到派发那一刻。

📜

dispatch contract — 一次锁定,逐字下发

Orchestrator 派发子任务时,把「必须互通的共享规格」(API、字段名、文件路径、端口、数据形状)写进 contract 字段。它被逐字交给每一个队友,也是验收他们交付物的依据。

contract: "金额一律用分(整数 amount_cents);
charge API = POST /v2/charge;
错误码遵循 RFC7807 problem+json"

在这里锁一次,别让每个队友各自发明自己的那套。还会在 orchestrator 收尾的 summary turn 里被重申一遍。

🧠

conv_memory — 全队共读的一份账本

共享记忆表 ConvMemoryRow,三种 kind:

  • contract — 锁定的共享规格
  • decision — 已拍板的决策
  • artifact — 产出物登记

Agent 用 remember 写、recall 读。冲突的来龙去脉(provenance:检测/解决)也写进来,orchestrator 收尾时读回、在 wrap-up 里提及 —— 闭环 + AI 协作记录两不误。

add_conv_memory / list_conv_memory · 跨 conv 还有 list_workspace_memory(全队)

🎯 这就是为什么意图冲突「不在合并期解决」: 合并引擎只能看见文本,看不见意图。把共享规格在动手之前注入(ADR-014/015 的 dispatch contract + conv_memory),意图冲突在它变成代码冲突之前就被消解了。剩下真撞车的,才落到上面的结构冲突闭环。
Undo Everything

回滚怎么做:四道「撤销」从轻到重

「可廉价回滚」是核心哲学的一半 —— 既然语义/意图冲突无法完备防住,那就让撤销足够便宜。Polynoia 有四层撤销,从「这次合并不算」到「整个 main 回到某个对话时的样子」。

⤺

① 守卫式 merge --abort — 探测不留痕

每次 probe_merge 撞冲突,立刻 git merge --abort 把 root 退回干净态。崩溃恢复 guard _abort_stray_merge 在每次合并入口检查:有 stray MERGE_HEAD 或 dirty index 就守卫式清理(先 rev-parse 验证再 abort,否则 rc=128)。root 永不停在半合并态。

sandbox/_core.py · _abort_stray_merge / git merge --abort(先 rev-parse -q --verify MERGE_HEAD)

✕

② abandon 冲突 — 这个分支不要了(但绝不静默)

解不动 / 不想解的冲突,点「放弃」→ POST /api/conflicts/{id}/abandon → status 翻 abandoned。卡上明示「已放弃,分支未合并进 main」+ 系统卡。决策被记录,不会像旧路径那样人间蒸发。

routes.py · abandon 端点 → ConflictRow.status='abandoned' + 系统卡

⊘

③ pending-edit 拒绝 — 改动还没落地就否决

manual 模式下每个写入挂在闸门上等审批。用户点 ✗ → MCP write 返回 {"error":"rejected by user"} envelope → 改动根本不发生、不 commit。这是最便宜的「回滚」—— 在它成为历史之前就拦下。

mcp/tools.py · _gate_via_pending_edit / _require_edit_approval

⟲

④ 「回到这个对话」— main 整体回滚 + undo 安全网

Cursor-checkpoint 风格:把 workspace main 硬重置到某个历史 commit。回滚前先把当前 HEAD 存进一个 undo ref(refs/polynoia/undo/<时间戳>)—— 旧 tip 不会丢,随时能再撤销回来。先 preview_restore_main(sha) 干运行(告诉你会撤掉哪些 commit / 文件 / 作者),确认了再 restore_main_to(sha) 真执行。全程持 workspace 合并锁、先 abort 任何半合并。

sandbox/_core.py · preview_restore_main(干运行)→ restore_main_to(reset --hard + undo ref)

回滚的「后悔药」:undo ref 双向安全网
📌 边界: 真正的「版本 checkpoint / restore 系统」(自动转换点快照、select-and-describe 片段修复)不在本设计,推 P2/P3。当前落地的是「回到某个 commit + undo ref」这一刀,够用且不引入复杂状态机。
What We Don't Solve

诚实边界:这套设计「不」解决什么

承袭深研版的诚实声明 —— 不夸大能力。明确说清覆盖缺口,比假装全包更可信(也更好答辩)。

① 不保证抓住所有语义冲突

文本合并成功 ≠ 语义正确(write skew 定理级漏检)。P1/P2 只验证「标记消除 + merge 能 conclude」,不验证编译 / 测试通过。语义验证推 P2+(Bayou 式重跑受影响测试),且动态语言覆盖有天然上限。

② 意图冲突不在合并期解决

由 dispatch contract + conv_memory 在动手之前传递共享规格(ADR-014/015)。本设计只处理「已经发生的结构冲突」,不在合并那一刻去猜两个 agent 当初想干嘛。

③ LLM 解冲突是辅助,不是保证

单次尝试,失败立即回落人工(Tricorder 10% 误报红线纪律:启发式信号只折叠提示、绝不阻塞)。真 git 冲突(确定性)才是 source of truth,LLM 只是「先试一下」。

④ 顺序合并的 stale 窗口

后解决的冲突针对当前 main(conclude_merge 重新快照 blob 缓解),但若先合并的兄弟分支后来被 revert,解决可能基于已变的 main —— 可接受,P1 以当前 main 为准。

理论锚点(为什么这样设计,不是拍脑袋): CALM / I-confluence(单调免协调)· SSI / write-skew(语义漏检定理)· Bayou(可执行断言=测试)· jj 一等冲突 + git rerere(冲突即可挂起的数据)· Tricorder(10% 误报红线)。

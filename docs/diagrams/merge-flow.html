Polynoia · Merge 双模式完整流程 v2 修正版

Manual = Cursor 式 per-edit 实时审批(每个 edit 出来就批) · Auto = Orchestrator 一人扛 · 切换字段 conversation.merge_mode
v2 修正了什么: Manual 模式不是"等 sub-task 全完了再 review 一整条分支",而是 每个 edit 出现就立刻推卡片给用户审批(Cursor / Windsurf 那种)。 另外:"悬挂"只悬挂该 tool call 这一个 coroutine,事件循环正常跑,其它 agent / 其它 conv / 同 agent 的其它 tool call 都不受影响。

Orchestrator Decompose 阶段

路径 A · Manual Per-edit 实时审批

3M · Sub-agents 进入各自分支开始干活
@claudeCode 在 agent/claudeCode/conv-A 上 · @codex 在 agent/codex/conv-A 上 · 并行
4M · @claudeCode 第 1 次 edit_file → MCP 悬挂这条 call
差 diff 推送给 UI,变成一张待审批卡:
invoice.py +2 −0 ⏳ 待审批
amount: Decimal + discount: Decimal = Decimal(0) created_at: datetime
✓ 接受 ✗ 拒绝
5M · 用户点击 ✓ 接受
WebSocket 把 decision="accept" 推回服务端 → 悬挂的 edit_file coroutine 醒过来 → 真的写文件 + commit 到 @claudeCode 分支 → 返回成功结果给 LLM,@claudeCode 继续推理下一步
6M · 多个 edit 串成一条审批流,中间用户可以拒绝
@claudeCode 下一个 edit_file → 又一张卡 → 用户审批 ...
@codex 那边同时也在出卡(独立队列)
如果用户拒:工具返回 {"error":"rejected by user"}, LLM 收到失败 → 重新想方案
7M · Sub-task 完成时,分支末端就是"用户已批准"的状态
因为每个 commit 都是用户点过的,直接 git merge --no-ff 进 main, 不需要再来一次 branch-level review
特点:用户在每个 edit 出现的瞬间就介入,粒度最细。 Sub-task 完成时分支已是"过审"状态,合并到 main 是水到渠成。

路径 B · Auto Orchestrator 全权

3A · Sub-agents 在各自分支飞奔
edit_file 直接写,不悬挂 · diff 卡推到 UI 但只是通知,无按钮 · 各自 commit 完结束 turn
4A · Orchestrator 进入 Merge 阶段
tools = [git_merge, git_status, read_file, edit_file, write, bash, ...]
收到所有完成分支的 commit 摘要 + main 当前 sha
5A · 自己跑 git_merge,顺序自己决定
git_merge agent/claudeCode/conv-A --no-ff → ✓ sha 111
git_merge agent/codex/conv-A --no-ff → ✗ conflict at invoice.py:42
6A · 撞了 → Orchestrator 自己解决,不派人
① read_file 看冲突标记 <<<<<
② 自己想清楚怎么合并
③ edit_file 写合并版
④ bash("git add -A && git commit")
⑤ 继续下一个分支
7A · 危险命令也是它说了算
git reset --hard / rm -rf / squash commit / force update branch
它觉得该干就干。没有"升级用户"逃生通道。
8A · 全完了 → 发结果卡
列每个分支合并状态 + Orchestrator 的判断解释 + main sha
特点:用户不介入,Orchestrator 全权。 它跟普通 specialist agent 拿同一套工具,身份在不同 phase 切换。

关键澄清:"悬挂"具体悬挂的是啥

Manual 模式下 edit_file 调用 await ctx.await_user_decision(...),只是把 这一个 coroutine 挂在 asyncio 事件循环上。事件循环上的其它任何 task 都正常跑。下面这张图是某一时刻的并发快照:

conv-A · @claudeCode
⏳ 等待 edit_file("invoice.py") 悬挂等用户点
conv-A · @claudeCode
▶ 跑 同一 turn 内并行发起的 grep("Invoice"),返回中
conv-A · @claudeCode
▶ 跑 read_file("models/__init__.py"),返回中
conv-A · @codex
▶ 跑 独立 subprocess,在自己分支改 currency 字段
conv-A · @designer
▶ 跑 在出色板,不受影响
conv-A · Orchestrator
▶ 跑 监控子任务进度,该派活继续派
conv-B
▶ 跑 完全无关的另一个对话,所有 agent 全速
apps/server FastAPI
▶ 跑 接收别的 WebSocket / HTTP 请求,毫无阻塞感
结论:"卡住"只卡住一个 await —— 该 tool call 后面那一步 LLM 推理可能要等用户。 系统资源、其它 agent、其它 conv,统统照跑。Polynoia 本来就是 multi-agent 多对话并发架构, per-edit 审批落到这套架构上是天然适配的。

两模式逐项对比

Manual(Cursor 式 per-edit) Auto(Orchestrator 全权)
edit_file 行为 悬挂等用户审批,过了才真写 立刻写文件 + commit
用户介入粒度 每个 edit 一次 不介入
Sub-task 完成时 分支末端 = 已批准状态,直接 merge Orchestrator 进 Merge 阶段
冲突谁解 通常不会发生(用户随时介入引导)。万一有,Orchestrator 同 auto 路径处理 or UI 介入 Orchestrator 自己 read + edit + commit
危险命令 每次都要用户点 Orchestrator 自己拍板,不挡
系统并发能力 同一架构,完全相同 — 单个 await 的悬挂不影响整体
切换方式 conv header 一键切 · 只影响未来 edit/merge

    • auto 模式:pool.get_session("orchestrator", conv_id, phase="merge") 给全套工具

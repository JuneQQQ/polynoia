1记忆系统

一张表 + 三种 kind,按场景两种读法,作为上下文装配的一层注入每一轮。

1.1　存储:一张表打天下　conv_memory

列说明

conv_id哪个会话(外键 → conversations,级联删)
author_agent_id谁记的
kindcontract | decision | artifact
content正文(1–2 行)
created_at时间

与聊天流水 MessageRow、workspace 级 PinRow 分开 —— 专门装"已锁定的共识 / 工作记录"。

1.2　三种 kind + 写入口(各有专路)

共享接口契约
✍ dispatch(contract=) 排空时自动种

全群拍板的决定
✍ remember(kind=decision) 工具

上一轮结论: remember 的 contract / artifact 两个枚举和 dispatch / report 重叠(而表是追加、不去重 → 双契约会打架)。建议收窄成 decision-only。

1.3　读取:两种形态,看场景(ADR-019 核心)

context/shared.py 按会话类型分叉。判定:workspace_id is None and (direct or members≤2) → 走 B,否则 A。

(A) 群聊 / 项目会话 → conv-scoped 共享板

list_conv_memory(conv_id) 取本会话全部,按 kind 分层渲染:

  • 契约 / 决策在前(完整)
  • 产物折成 headline(G-Memory 分层的最廉价近似)

(B) 项目外单聊 → agent-level 记忆

  • ## 我的工作 — list_agent_memory(author) 跨所有会话自己记的
  • ## 队友相关工作 — list_workspace_memory(ws) JOIN 会话表取项目内别人记的(压缩)
  • 配合只读挂载 → 能"真读代码"核对细节
⚠ 隐患仍在: list_agent_memory 没有隐私过滤,纯按 author_agent_id 跨所有会话捞 → 跨会话 / 跨用户可能泄露。P1 多用户前要按 workspace 成员资格过滤(同 L3 ledger)。

1.4　记忆活在更大的"上下文装配"里(L1–L5)

context/assembler.py 按固定顺序拼六层,\n\n---\n\n 隔开。记忆是其中的 L2.5。

L1# 身份 identity 你是谁 + 平台@协议 + persona prio 100HARD
L2# 你的项目 brief 所在 workspaceprio 70
L3# 跨对话活动 ledger 跨会话事件(带隐私过滤)prio ~
L2.5<shared_memory> 　← 上面讲的记忆(A 或 B) contract / decision / artifactprio 55
L4# 本对话历史 history 最近 30 条,带 [@名字]prio 60
L5# 当前用户消息 user_turn 触发本轮的文本 prio 90HARD

priority 只管"预算不够先裁谁"(ledger→history→brief→shared 先裁;hard 永不裁)。
预算(budget.py):模型上限 − ~35k 开销;ADR-019 修了"shared 层无上限"的 bug,从 history 切 0.08 给它。
主动读: 除被动注入,agent 还能中途调 recall 即时查共享记忆(不等下一轮)。

2编排系统　工具驱动 · boomerang

无专用代码 —— 靠"一个普通 agent 轮 + 它手里的 dispatch 工具"完成。

2.1　入口路由　dispatch_user_message(routes.py:2440)

conv.orchestrator_member_id 指定了某成员? ├─ 是 → run_adapter_turn(orch_id, text, is_dispatcher=True) # 工具编排 └─ 否 → 普通群聊: ├─ 文本里有 @ → 只派给被 @ 的成员(Slack 语义) └─ 无 @ → 扇出给所有 adapter 成员

没有隐式 orchestrator —— 不指定就是平铺群聊。

2.2　boomerang 全流程

orchestrator 的 run_adapter_turn 跑起来

session 带 role-scoped dispatch 工具。

它调 dispatch(title, contract, tasks=[{agent,note}…])

声明这一批要并行干的活。

/dispatch 端点 stash 进 _pending_dispatches[conv_id]

注意:不立刻执行。

orchestrator 这一轮文本结束

run_adapter_turn 在轮尾 DRAIN

排空 _pending_dispatches,做三件事 ↓

├─ 发一张 BurstCard(并行泳道,N 个子任务)
├─ 把 contract 自动种进 shared memory
└─ 为每个 task 并发 spawn 一个 worker run_adapter_turn,worker_text =
　"# 接口契约(锁定)\n{contract}\n\n# 你的子任务\n{note}
　 \n\n# 收尾(必须)调用 report 自评 + 用 recall 查共享记忆"

每个 worker 干完翻自己泳道状态

done / failed。

_mark_burst_task 做 barrier

谁是最后一个完成的,同步"认领 last" → 触发:

├─ 静默 merge to main(auto 模式 + workspace)
└─ orchestrator 自动汇总轮(suppress_dispatch=True,不再派活)
🪃 形象点: 派活像回旋镖扔出去(fire-and-forget),worker 各自干完飞回来,最后一个回来的人负责收尾 + 叫 orchestrator 总结。

2.3　关键机制

闭环交接(ruflo/ADR-014):worker 必须 report 自评(status / deliverables / contract_ok),orchestrator 汇总轮读回 verdict 核对,而非"沉默即成功"。
partial-success:某 worker 挂了不 cancel 全部,标 failed,其他继续。
1-shot 重试:瞬时失败重排一次(max_attempts)。
深度限制:@接力链有 _MAX_MENTION_CHAIN_DEPTH,防无限派发。
merge phase:auto + workspace,各 agent/*/conv-* 分支 git merge --no-ff 进 main,冲突标 needs-manual。
身份门控:is_dispatcher 按"轮的身份"标记,abort 时只清自己那批 pending,不误伤兄弟泳道。

3工具　MCP · mcp/tools.py

3.1　工具清单

文件 / 沙箱类(干活):

readeditwriteapply_patch bashgrepglobrevert

协作类(多 agent 的胶水):

工具干啥给谁
dispatch派并行子任务 + 锁契约只 orchestrator
reportworker 收尾自评(闭环交接)→ 写 artifactworker

三句话总结

  • 记忆 = 一张 conv_memory 表 + 三种 kind,按场景两种读法(群聊共享板 / 单聊 agent-level),作为 L2.5 层注入每一轮。
  • 编排 = 无专用代码,靠"orchestrator 成员 + dispatch 工具 + 轮尾 drain + barrier"的 boomerang 闭环。
  • 工具 = 文件类 + 协作类(dispatch / report / remember / recall / ask_user),按角色和位置双重门控。

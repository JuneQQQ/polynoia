Polynoia · Credential Design

跨平台 Agent 凭证策略

Polynoia 需要同时启动 Claude Code、Codex、OpenCode 等 agent CLI。设计目标是:隔离每个 agent 的配置与 MCP 注入,同时复用用户本机已经登录过的账号,并尽量不复制、不暴露、不污染真实凭证目录。

00核心结论

凭证策略分成两层:真实登录凭证尽量复用宿主环境;运行期配置、MCP server 注入、agent 临时状态放入 Polynoia sandbox。

macOS 本机默认使用 direct 模式,因为 Claude Code 的 OAuth 登录态可能在系统钥匙串里,强行改 HOME 反而会让 CLI 找不到登录态。Linux、Windows、root/server/container 环境默认使用 sandbox-copy 模式,只复制 allowlist 内的运行凭证文件到本地隔离目录。

平台策略 · 适配器策略 · 运行流 · Linux 与 Codex 凭证 · 安全规则 · 环境变量 · 代码锚点

01平台策略

环境默认模式理由处理方式
macOS local direct Claude Code 登录态可能由 Keychain 管理,不是单纯的文件复制。直接使用宿主 HOME 更符合本机桌面使用方式。 Claude/OpenCode 优先使用宿主凭证;Codex 仍使用独立 CODEX_HOME,只同步必要文件后注入 Polynoia MCP。
Linux / server / container sandbox-copy 通常没有 Keychain 这类桌面凭证服务,且服务进程、root、容器用户与真实登录用户经常不一致。 从 POLYNOIA_CRED_HOME 或真实用户 home 复制 allowlist 文件到 sandbox,再重写 HOME/XDG_*。
Windows sandbox-copy 凭证和应用数据分散在 USERPROFILE/APPDATA/LOCALAPPDATA,不应假设 macOS Keychain 语义。 为 agent 设置独立的 profile/appdata 根目录,复制必要 auth/config/database 文件,避免污染用户真实配置。

02适配器策略

Adapter凭证处理MCP 注入设计理由
Claude Code macOS direct:使用真实 ~/.claude 与 Keychain 项 Claude Code-credentials。非 direct:复制 .claude allowlist 到 sandbox。 通过 SDK 参数传入 mcp_servers={"polynoia": ...}。 Claude 的登录态不总在文件系统里。优先让官方 CLI 按原生方式找凭证,减少脆弱的跨平台复制逻辑。
Codex 始终使用 sandbox CODEX_HOME。每次会话前同步 config.toml、auth.json、模型缓存等 allowlist 文件。 写入 [mcp_servers.polynoia] 与 [apps.polynoia] 配置块。 Codex 需要改配置才能挂 MCP 与 app-server 权限策略。使用独立 CODEX_HOME 可以注入配置,但不改用户真实 ~/.codex/config.toml。
OpenCode 使用隔离的 XDG_DATA_HOME/OPENCODE_DATA_HOME,从宿主数据目录复制或引用登录数据。 通过 ACP session/new 传入 mcpServers。 OpenCode 的运行数据可能涉及 SQLite/WAL 文件。隔离数据根目录能降低与用户交互式 OpenCode 抢锁或互相污染的概率。

03运行流

macOS Claude direct

agent process→ real HOME→ ~/.claude+ Keychain

不重写 HOME,避免把 Claude 从系统登录态里切出去。

Linux sandbox-copy

host credential home→ allowlist copy→ .polynoia/credentials→ HOME rewrite

服务端、容器、root 场景更可控,也更容易清理。

Codex CODEX_HOME

~/.codex→ sync→ sandbox/.codex→ merge MCP config→ codex app-server

这是配置隔离和工具接管的关键路径。

04为什么 Linux 可以复制 Codex 凭证,但不能随便改

Linux 上复制 Codex 的 auth.json 和 config.toml 是可行的,因为没有 macOS Keychain 这类额外依赖。但它仍然应该被当成易过期的 OAuth 快照,不是长期稳定的系统凭证。

  • 可以复制:复制到 sandbox CODEX_HOME,再在副本里注入 Polynoia MCP 与 app 配置。
  • 不要改原件:真实 ~/.codex/config.toml 属于用户自己的 Codex 环境,Polynoia 不应把 MCP、审批策略、实验开关写回去。
  • 需要刷新:会话启动前同步一次;遇到 401/登录失效时重新同步或提示用户重新登录。
  • 更稳定的未来形态:如果 Codex 支持 auth-home 与 config-home 分离,Polynoia 可以直接引用官方 auth store,只隔离配置目录。

05安全与运维规则

规则说明
allowlist only只复制明确列出的运行凭证/配置文件,不递归复制整个 home。
never commit.polynoia/credentials、sandbox auth 目录必须保持本地私有,不能进入 git。
no secret UI/logUI、日志、错误提示只展示路径类别和状态,不打印 token、refresh token、完整 auth JSON。
refresh per session每次 agent 会话开始前同步一次,避免使用很久以前的过期快照。
clear on reset测试重置、项目清理时可以删除 sandbox copy,真实宿主凭证不受影响。

06环境变量开关

POLYNOIA_DIRECT_CREDS=1      # 强制 direct,优先使用宿主凭证环境
POLYNOIA_DIRECT_CREDS=0      # 强制 sandbox-copy
POLYNOIA_CRED_HOME=/Users/me # 指定真实凭证来源 home,适合 sudo/root/server
CODEX_HOME=.../sandbox/.codex # Codex adapter 设置,不要让模型直接写真实 ~/.codex

默认值按平台和运行身份选择:macOS 非 root 优先 direct;其他环境优先 sandbox-copy。显式环境变量永远优先。

07当前代码锚点

文件职责
apps/server/polynoia/credentials.py统一判断 direct / sandbox-copy、解析凭证来源 home、同步 Codex allowlist 文件。
apps/server/polynoia/sandbox/_core.py为 sandbox 准备跨平台凭证目录与兼容旧逻辑的凭证入口。
apps/server/polynoia/adapters/codex.py创建 sandbox CODEX_HOME,同步 Codex 凭证,注入 mcp_servers.polynoia 与 apps.polynoia。
apps/server/polynoia/adapters/claude_code.pymacOS direct 模式使用系统 Claude CLI 与 Keychain 登录态;其他模式走 sandbox 凭证。
apps/server/polynoia/adapters/opencode.py把 OpenCode 数据目录与真实用户 home 解耦,避免锁冲突和配置污染。
本文档描述当前 Polynoia agent CLI 凭证隔离策略。原则:能走官方登录态就不复制;必须注入配置时只复制副本;跨平台行为显式、可覆盖、可清理。

② 谁能用什么:角色不发明工具差异,只映射到几个「层」

关键设计:角色的工具不是逐个手列的,而是从几条能力轴组合出来的,角色名只是映射到几个功能层。 persona 的差异(码农 vs 设计 vs 文案)活在 System Prompt 里(它擅长什么),不在工具集里(它能碰什么)。 真正分化的轴只有 4 条:能否改文件(write) · 能否开 shell(bash) · 编排者(dispatch/present) · worker(report)。

7 个角色名 → 5 个工具层(coder≡generalist、designer≡writer 逐字相同,差异全在 prompt):
  • orchestrator —— 建造 + dispatch/discuss/present,无 report(它读别人的 verdict,不自评)
  • builder(coder · generalist)—— 全套建造:retrieve + 记忆 + ask + write + bash + worker(report/req_access)
  • builder−shell(designer · writer)—— 同 builder 但无 bash(逼显式 write 落产物)
  • consult(advisory)—— 只读 + 记忆 + ask + report;无 write/bash(主页 DM 降级态)
  • auditor(critic)—— 最小只读:retrieve + recall + report 一个 verdict,连 remember 都没有

为什么这样分(而不是「7 个角色 7 套工具」):
  • 建造型角色本就该同一套宽工具 —— coder/designer/writer/generalist 的差别全在 prompt(人设/技能),工具上只剩 bash 这一条轴(designer/writer 没有,逼显式 write 落产物)。所以它们塌缩成 2 个层,不是 4 套。
  • 工具该分化的只有「安全」和「结构」 —— 安全 = 能不能写(只读的 critic/advisory vs 可写);结构 = 编排者(dispatch/present、无 report)vs worker(report、无 present)。这两条是承重的,其余都不该按角色硬分。
  • per-agent 覆盖才是细粒度入口 —— 用户在「高级」里勾的 tools_whitelist 会在角色层上再收窄(role ∩ allow,只减不增)。所以层是默认值,真正的「每 agent 工具集」(rule.md §自建 Agent)靠这层 whitelist。
  • advisory 不是 persona,是降级态:对话不属于任何项目时(主页 DM),适配器池把任何联系人降到只读咨询 —— 没有共享沙箱可写。

③ write 的「人在回路」门控(manual 模式)

开了 manual 合并模式时,每次 write 不直接落盘,先挂一张 pending-edit 卡等用户点 ✓ / ✗:

1
agent 调 write → MCP 工具进程先查 conv.merge_mode
tools.py · _gate_via_pending_edit
2
auto 模式 → 直接放行落盘(无门控)。manual 模式 → POST /api/pending-edits 建一张待审卡 + WS 广播给前端
3
阻塞轮询 /pending-edits/{id}/wait,直到用户在 UI 上拍板
✓ 接受 → write 真正落盘 + commit,工具返回成功,agent 继续
✗ 拒绝 → 不落盘,工具返回「被拒」,agent 据此调整
关键:门控在 MCP 工具进程里做(不是 adapter 层)—— 对模型透明,它只看到「write 成功 / 被拒」。 冲突闭环(feature/diff_dev)正逐一镜像这条 pending-edit 轨道做「合并冲突选边」。

Agent 工具调用设计

Polynoia 经 MCP(stdio)暴露给每个 agent 的 14 个工具:它们的入参 / 出参、跨工具的执行超时与人审批机制、以及按角色的可见性。本页描述当前实现(源:apps/server/polynoia/mcp/tools.py + server.py)。

00一次工具调用怎么走

Adapter(Claude Code / Codex / OpenCode)在 agent 的 turn 里发出 tool_use → Polynoia 的 MCP server(每个 agent session 一个 stdio 子进程)收到 call_tool(name, arguments) → 修复参数 → 按角色校验 → 套执行超时跑 tool.execute(ctx, args) → 返回结构化结果给模型。

adapter tool_use→ _repair_arguments→ role 校验 (ROLE_TOOLS)→ _execute_bounded(超时)→ execute → 结构化结果→ 模型读到出参

所有出参都是一个 dict,常带 kind 判别字段;失败统一为 {"kind":"error","error":...}(或 {"error":...})。沙箱在 ~/sandbox/<conv>/…/worktrees/<agent>,文件类工具受沙箱限制 + 走 git。

01跨工具机制(所有工具共享)

执行超时(_execute_bounded @ server.py)

  • 默认 60s(_DEFAULT_TOOL_TIMEOUT):每个工具调用都有界,超时不再卡死整轮,而是把超时结果反馈给 agent,它可重试或换路。
  • agent 可覆盖:任意调用传 timeout(秒)即可调整。
  • 超时出参:{"error":"⏱ 工具 X 执行超过 Ns…","timed_out":true,"tool":X,"timeout_s":N}。
  • 两类豁免:human_wait 等人工具(ask_user / request_project_access / write·edit 的审批门)不计时——它们在人侧已有上限;self_timeout 的 bash 自管超时(默认 30s)+ 优雅杀子进程,包装仅作宽限兜底。

人审批门(manual 合并模式 · write/edit)

manual 模式下,write/edit 先经 _require_edit_approval long-poll 等用户批准(审批卡渲染该次改动的 diff)。被拒 → 返回 {"kind":"rejected","error":"rejected by user"},不落盘。auto 模式直接执行。

按角色可见(ROLE_TOOLS,能力轴组合)

角色可用工具
orchestrator全部读写 + dispatch / discuss / present / resolve_conflict(协调专属);无 report
coder / generalistread / write / edit / bash / grep / glob + recall/remember + ask_user + report + request_project_access
designer / writer同上但无 bash(强制走 write/edit)
critic只读:read / grep / glob / recall + report
advisory只读 + ask_user / remember / request_project_access(无 write/edit/bash)

能力轴:_RETRIEVE{read,grep,glob} · _MUTATE{write,edit} · _SHELL{bash} · _ORCHESTRATE{dispatch,discuss,present,resolve_conflict} · _WORKER{report,request_project_access} · _RECALL/_REMEMBER/_ASK。角色差异在系统提示,不在工具集 —— coder≡generalist、designer≡writer(同层)。

02逐个工具:入参 / 出参

read · write · edit · bash · grep · glob · dispatch · discuss · present · resolve_conflict · remember · recall · report · ask_user · request_project_access
文件 · 读写编辑搜索

read

读沙箱内 UTF-8 文本文件,返回带行号的内容;大文件加固:逐行流式(不整读)、单行截断 2000 字、单次 ≤50KB、超出给续读游标。
入参
参数类型说明
path *string相对沙箱根或沙箱内绝对路径
offsetint ≥1起始行(1-indexed),默认 1
limitint ≥1最多返回行数,默认 2000
出参
{kind:"file", path, content,            // content = 带行号 "  N→…"
 total_lines, returned_lines,
 next_offset?, hint?,                   // 还有更多内容时给续读游标
 output_truncated?}                     // 因 50KB 上限截断
{kind:"directory", entries:[…]}         // path 是目录时
{error:"file not found: …"}

write human_wait·审批

新建或整体覆盖文件,自动建父目录 + git 提交。改局部应优先 edit(整文件重写慢且易误删)。>10MB 拒绝。
入参
参数类型说明
path *string目标路径
content *string完整文件内容
turn_idstring归属本轮(提交信息用)
出参
{kind:"wrote", path, created:bool, bytes, commit_sha}
{kind:"error", error}            // 超 10MB 等
{kind:"rejected", error}         // manual 模式被用户拒绝

edit ★ 大文件主路径 human_wait·审批

对已有文件做定向 old_string→new_string 替换(改动成本正比于改动量,不是文件大小)。old_string 须逐字匹配且唯一(否则失败要更多上下文),或 replace_all 全替。新建用 write。
入参
参数类型说明
path *string已有文件
old_string *string要替换的原文(逐字,含缩进;不要带行号前缀)
new_string *string替换为
replace_allbool替换全部匹配,默认 false
turn_idstring归属本轮
出参
{kind:"edited", path, replacements:int, commit_sha}
{kind:"error", error}     // 未找到 / 命中多处需 replace_all / 空串 / old==new / 文件不存在
{kind:"rejected", error}  // manual 模式被拒

前端 diffUnified.ts 已原生支持 kind:"edit",审批卡 / diff 评审直接渲染。

bash self_timeout

在沙箱工作目录跑 shell,流式输出到终端卡。自带超时(默认 30s,可传 timeout)+ 优雅杀子进程。禁 pkill/killall、kill -1/-<pgid>(共享宿主进程空间,会被拦)。不 git 提交。
入参
参数类型说明
command *stringshell 命令
timeoutnumber秒,默认 30
出参
{kind:"completed", stdout, stderr, exit_code}
{kind:"timeout", timeout_s, …}          // 超自身 timeout
{kind:"blocked", command, reason}        // 命中宿主安全护栏

grep

沙箱内递归 grep(ripgrep 语义),返回 file:line。命中超 200 截断。
入参
参数类型说明
pattern *stringripgrep 正则
pathstring子目录,默认 "."
globstring文件名 glob 过滤
出参
{kind:"results", matches:[ "file:line: …" ], total}
{kind:"truncated", matches, total:200}     // 命中过多

glob

按 glob 找文件;超 500 截断并给标志。
入参
参数类型说明
pattern *string如 **/*.py / src/**/*.ts
出参
{kind:"results", paths:[…(≤500)], total, truncated:bool}
协作 · 编排(orchestrator 专属)

dispatch

把工作并行派给队友(各自 worktree 执行、合回 main)。立即返回(fire-and-forget),队友工作以并行泳道流入对话。多阶段计划在非末批设 need_continue:true 以自动推进。
入参
参数类型说明
tasks *array每项 {agent, label?, note};note 是给队友的完整自包含 prompt(纯文本,勿嵌 JSON)
contractstring所有子任务须遵守的共享契约(接口/字段/路由/端口),逐字注入每个队友并用于验收
titlestring批次标题(UI 卡头)
need_continuebool非末批设 true → 平台给后续轮可继续 dispatch;末批 false → 后续轮只验收+present
出参
服务器 ack(经 /dispatch):批次 task_ids 等;失败 {kind:"error", error}

discuss

发起队友间自由讨论(非并行干活):权衡选项/评审设计/达成共识。≥2 名参与者,自动收敛并产出单条「讨论结论」。立即返回。
入参
参数类型说明
topic *string讨论什么(纯文本)
participants *array≥2 名队友显示名
出参
服务器 ack;失败 {kind:"error", error}

present

把面向用户的交付物聚成一个面板(预览/下载)。只 present 用户真正消费的东西(可运行 HTML、文档、图片),不要 present 代码源码树(diff 卡已逐文件展示)。一次调用。
入参
参数类型说明
pathsarray多个工作区相对路径,聚成一个面板
pathstring单个路径(paths/path 二选一)
messagestring交付一句话说明(显示在文件上方)
出参
{presented:true, paths:[…]}
{presented:false, deferred?/note?/error?}
{error:"path or paths required" / "file not found: …"}

resolve_conflict orchestrator-only

auto 模式下,队友分支合并冲突出现时,由 orchestrator(中立仲裁者,握有契约+各方意图)逐文件决断并合入 main。须覆盖每个冲突文件,否则合并中止。
入参(至少给一种决断)
参数类型说明
conflict_idstring冲突 id(只有一个开放冲突时可省)
resolutions{path→text}完整合并后文件文本(去掉所有冲突标记)
sides{path→'ours'|'theirs'}整边采用(ours=main,theirs=队友分支)
deletions[path]删除该文件(改/删冲突)
出参
{resolved:true, sha, conflict_id}
{resolved:false, error/note}
共享记忆 (ADR-014)

remember

把一条事实写入对话共享记忆(每个队友每轮都读到):decision / artifact / contract。每条 1-2 行,记决策或接口,别贴实现。
入参
参数类型说明
content *string事实,1-2 行
kindenumdecision / artifact / contract
出参
服务器 ack;失败 {kind:"error", error}

recall

立刻读对话共享记忆(契约/决策/产物),可按 kind 过滤。用于任务中途核对契约是否变了 / 队友交了什么。
入参
参数类型说明
kindenum可选过滤:decision / artifact / contract;省略=全部
出参
服务器返回记忆条目列表
人在回路 / 汇报

report

子任务结束时向 orchestrator 闭环汇报:交付了什么、是否符合契约、有何遗留。诚实给 ok/partial/failed,显示在泳道、抗刷新。
入参
参数类型说明
status *enumok / partial / failed
deliverables *string实际产出:文件名 + 一行说明
contract_okbool是否符合共享契约
notesstring(可选)caveat / 风险 / 缺失
出参
服务器 ack;失败 {kind:"error", error}

ask_user human_wait

向用户提问并挂起本轮直到回答,在同一轮拿到决定继续(优于猜测或写"等用户指令")。≤4 题/次。
入参
参数类型说明
questions *array1-4 题,每题 {id, label, kind:'single'|'multi'|'fill', options?, optional?, placeholder?}
titlestring问题卡标题
出参
{kind:"answered", answer:"…"}
{kind:"error", error}

request_project_access human_wait

私聊 1:1 中,用户让你做某项目时申请其项目访问。挂起本轮 → 用户审批卡(选项目 + 批准/拒绝)。批准后下一轮挂载写权限。
入参
参数类型说明
reason *string为何需要访问 / 打算做什么
出参
{kind:"granted", workspace_id}
{kind:"denied"}
{kind:"error", error}

03并行工具调用

支持。Adapter 可在一条 assistant 消息里发出多个 tool_use;MCP server 的 call_tool 对每个独立处理,asyncio 下并发执行(无全局工具锁)。安全边界靠:同文件 ctx.file_lock 串行化、同 agent 同文件不会丢更新;不同工具/不同文件并发跑。所以一个 agent 可以同时跑两条 bash(例如一条 npm i + 一条看门狗),两张终端卡都显示运行中。

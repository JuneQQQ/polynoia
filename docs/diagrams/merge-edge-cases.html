合并 / 同步管线 —— 边界情况与处理

Polynoia 的每个 agent 在 workspace 的独立 git worktree 里干活,产出经「单一合并咽喉」_drain_unmerged_branches 汇入 main。 这份文档梳理了这条管线上 5 个被点名的潜在 bug —— 哪些是真的、怎么修的、哪个是虚惊 —— 外加分支同步从 merge 改 reset --hard 的重做。
scope: apps/server/polynoia/{sandbox/_core.py, api/routes.py} · 冲突闭环承重区 (CHARTER) · tests: tests/sandbox/test_merge_helpers.py
已修 / 安全 真 bug(数据丢失 / 错乱) 设计取舍 / 边界守卫 重做(同步模型)

合并咽喉:一次 drain 的五步

burst 全部完成 / 单个 turn 结束时各触发一次,全程持 workspace_merge_lock(ws_id)(per-workspace,非 per-conv)。

step 1
commit_pending_worktrees
把 agent 用原生工具写、还没 git commit 的改动收成 commit。仅扫 owner_agents。
step 2
list / ahead 过滤
遍历 agent/*/conv-X 分支,跳过 0-ahead 和已有冲突卡的。
step 3
probe_merge(b)
试合并:clean→直接 commit;conflict/error→--abort,main 不动。
step 4
files_in_range
pre_main..post_main 算出本分支落地的文件,挑出可预览的当 deliverable。
step 5
handoff / cards
有 deliverable 或冲突 → 交给 orchestrator 展示;solo 会话直接发 file 卡。

下面每个边界情况都标注它落在哪一步。橙色步骤(1、3)是这次改动的重点。

#1 真 bug半成品被别人的 drain 提交进 main · step 1

旧的 commit_pending_worktrees(conv_id) 对本会话所有 worktree 无脑 git add -A && commit。 但 drain 可能由 另一个 agent 触发 —— 此时第三个 agent 还在写文件。

之前 · 会污染 main

# 苏念 还在写 onepager.md(未 commit)
# 顾屿 turn 结束 → 触发 drain
commit_pending_worktrees(conv)
  → sweep 苏念 worktree 也被 add -A
  → 苏念 半成品 commit 到它的分支
  → 一起 merge 进 main  ❌

之后 · owner 作用域

commit_pending_worktrees(conv, only_agents)
  # 单 turn drain → owner = {当前 agent}
  # burst drain  → owner = {本批 workers}
  → 只扫 owner 的 worktree
  → 苏念 (mid-turn) 被跳过,半成品保留 ✅
关键点: 跳过非 owner 不会漏合并任何已完成的工作 —— 别人已 commit 的分支照样在 step 2 被 ahead 检测捞到合并;被跳过的只是未 commit的进行中改动,等那个 agent 自己 turn 结束再收。
owner 来源:单 turn = {agent_id};burst = {t["agent"] for t in reg["payload"]["tasks"]}。None = 扫全部(无并发时的兜底)。测试:test_commit_pending_worktrees_only_owner。

#2 取舍probe_merge 是「探测即落地」 · step 3

名字叫 probe,但 clean 时它真的 commit 到 main(冲突闭环靠它做唯一合并)。这是有意设计,不是 bug —— 它对单个分支是原子的:

分支结果对 main 的影响恢复
cleanmerge commit 落地(--no-ff)不需要
conflict不动(git merge --abort)出冲突卡,分支保留待解
error不动(_abort_stray_merge)分支留在 ahead,下次 drain 再试

真正的隐患在调用方:若一个分支 clean 落地后、记账(files_in_range / 发卡)抛异常,会把整个循环带崩,后面分支的 deliverable / 冲突信号全丢。

加固 · 每分支隔离 + 提前推进 pre_main

for b in branches:
    try:
        status, detail = probe_merge(b)
        if status == "clean":
            prev_main, pre_main = pre_main, post_main   # 先推进,再记账
            changed = files_in_range(prev_main, post_main)
            ...                                  # 抛了也不会错配下个分支
    except Exception:
        log.exception(...); continue            # 一个分支崩 ≠ 整批崩

#3 虚惊跨冲突分支的 pre_main 追踪 —— 其实是对的 · step 4

担心:A 合了(pre_main 推进)、B 冲突了(pre_main 没推进)、C 再合时把 A 的文件混进 C 的 deliverable。核实后:不会。 

# 关键事实:冲突会 --abort,main 不前进
pre_main = M0
A clean  → main = M1   range(M0,M1)=A 的文件   pre_main = M1
B conflict → main = M1 (aborted,没动)            pre_main = M1 不变  ✓
C clean  → main = M2   range(M1,M2)=只有 C 的文件  ✓

正因为冲突不推进 main,「冲突后不更新 pre_main」恰恰是正确的 —— C 的 range 起点仍是上一次真实的 main 状态(M1),B 没贡献任何 main 变更可混入。无需改动,这里把不变量记录下来防回归。

#4 潜伏去重漏了 present 的 kind:files 面板 · step 5

自动 file 卡发 kind:"file"(单数);orchestrator 的 present 发 kind:"files"(复数,一个面板含多文件)。 旧去重只扫单数 → 同名文件可能既进 present 面板又被自动补一张卡。

当前现状

自动卡只在 solo/direct 会话发(auto_cards = not group);present 只在群聊由 orchestrator 用。两者按会话类型互斥,所以现在不会真撞上。

加固 · 去重覆盖两种 kind

if kind == "file":  seen.add((sid, name))
elif kind == "files":                # present 面板
    for f in payload["files"]:
        seen.add((sid, f["name"]))

为什么还是修:那条「已被 present 去重」的注释本来就是错的;且一旦门控变化(比如 solo 会话也允许 present)立刻变真 bug。修在去重层(对所有历史曝光去重)是更对的高度,而非依赖门控巧合。

#5 边界git checkout main 在 worktree 环境

merge_branch_into_main / probe_merge 都在 workspace root 上 git checkout main。 git 禁止两个 worktree 同时 checkout 同一分支 —— 若某个 agent worktree 异常 checkout 了 main,root 的 checkout 会被拒。

正常态下每个 agent worktree 永远在自己的 agent/*/conv-* 分支,只有 root 持有 main,所以不会触发。 现有保护已足够:checkout 失败会返回 ("error", …) 而非崩溃,且 _abort_stray_merge 先清理半合并残留。判定:真实但极罕见,记录为已知边界,不额外加码(加码反而增加承重区复杂度)。

重做分支同步:merge main → reset --hard main

一次任务后 main 是「解过冲突」的代码。下次对话 agent 分支若用 git merge main 同步,会重放已解决的冲突、并保留被拒的旧版本 → 偏差累积。 洞察:agent 分支是一次性工作副本 —— 上一轮产出要么已 merge 进 main、要么已被拒,没有独立价值。

之前 · merge,且漏 pooled

  • 同步藏在 create_workspace_sandbox 里,只在 reused 时跑
  • pooled session 复用根本不进这条路 → 不同步(靠 burst 末尾 evict orchestrator 这种打补丁)
  • dirty 就 skip;merge 冲突就 abort → 同步经常没生效

之后 · reset,每 turn,全覆盖

  • git reset --hard main —— 永不失败,丢弃偏差,从 main 起步
  • 提到 run_adapter_turn 入口,每个 turn 开始都跑(含 pooled 复用)
  • 在 worktree 里重定位自己的分支 ref,不 checkout main,与 root 安全共存
  • 退役了 burst 末尾那条 evict orchestrator 的补丁

⚠️ 唯一的守卫:未解决的冲突分支不能 reset

一个分支若有未解决的冲突卡,它那一版只活在分支上,等用户选边。此时 reset --hard main 会把用户还没选的那一版销毁 → 冲突闭环断裂。所以加门控:

turn 开始时,分支状态动作理由
干净 / 已合并 / 有被拒残留reset --hard main无独立价值,从 main 重新开始
有 open / resolving 冲突卡跳过同步待解的那一版只在分支上,不能丢
worktree 还不存在(全新 agent)no-op首次 spawn 时从 main 切出,本就最新
未 commit 残留(上轮崩了没 drain)reset 丢弃没进 main 的就是要丢的(用户已认可)
# run_adapter_turn 入口(持 agent 锁,与本 agent 其它 turn 串行)
has_open_conflict = any(r.branch == f"agent/{agent_id}/conv-{conv_id}"
                       and r.status in ("open", "resolving")
                       for r in list_conflicts(conv_id))
if ws_id and not has_open_conflict:
    await Sandbox.reset_worktree_to_main(ws_id, conv_id, agent_id)
守卫放在 routes 层(能查 DB 冲突表);git 动作放在 _core 层。reset 是唯一调用点,所有破坏性 reset 都在门控之后。测试:test_reset_worktree_to_main_discards_divergence / _noop_when_absent。

同步时机:为什么在「下次 turn 开始」而不是「orchestrator 总结后」

t0
任务结束
main = 解过冲突的代码。各 agent 分支已 merge 或被拒。
t1
(此期间)
用户继续聊;main 可能又被改;某 agent 可能根本不被再调用。
t2
下次 turn 开始
才 reset —— 拿到的是此刻最新的 main,且只对真要干活的 agent 做。

「总结后立刻同步所有人」会同步到一个可能过期的 main,还会浪费给那些再也不被调用的 agent。turn 开始前同步是惰性的、永远最新的。

Polynoia · 上下文 · 记忆 · 编排 · 工具

每次 agent 回复前，平台如何把散落的信息拼成 prompt —— 以及记忆、boomerang 编排、MCP 工具门控如何协同

L1–L9 上下文 · 记忆 · 编排 boomerang · 工具门控 · 3 种场景

1 全景：L1–L9 上下文 + 预算窗口

L1身份 IdentityHARD你是谁、后端、工具纪律、人格设定
L2协调器协议 / 群成员名单条件群聊专属：强制派活指令 + 可派成员；普通成员拿到可@名单
L4项目简报soft当前项目名/仓库/成员 + 你也属于哪些项目
L5共享记忆soft群聊：锁定的契约/决策/产物；私聊：agent 工作记忆
L6活动账本soft跨对话最近动态：其他对话里的消息 + 代码提交
L7对话历史soft当前对话最近 N 条消息（滚动窗口）
L8置顶消息soft用户手动 pin 的关键信息，长期上下文
L9用户当前消息HARD触发本轮回复的用户原文——截断它 = 答非所问

2 组装流程

1 定位 agent — DB 取 Agent 行
2 逐层构建 — L1→L2→L4→L5→L6→L7→L8→L9
3 计算预算 — 模型上下文上限 − 适配器开销（~35k），按比例分配
4 预算执行 — 2-pass：锁 hard 层；soft 层按优先级分配，超限截断
5 拼接输出 — 各层用 \n\n---\n\n 连接

3 各层详解

L1 · 身份 HARD · p100

context/identity.py → build_identity_layer

  • 名字、handle、id、后端适配器、模型
  • 平台简介 — Polynoia 是什么、IM 形态多 agent 协作
  • 工具纪律 — 按 tool_role 注入「能力边界 + 使用规矩」，行为化、不枚举工具名：工具 schema 已由 MCP 注入到请求 tools 字段，再列一遍是冗余。ask_user/dispatch/discuss 以「有就用，没有就忽略」条件写法并入
  • 项目内职责 — 仅群聊项目对话才注入 member_role
  • 人格/工作风格 — 用户编辑的 system_prompt（纯人格：身份/领域/语气/产物语言；工具纪律交给平台）

# 身份 你是 **顾屿**(handle:`@顾屿`,id:`01JX...`)。 由 `claudeCode` 后端驱动,model = `claude-sonnet-4-6`。 ## 关于平台 你跑在 **Polynoia** — 一个 IM 形态的多 agent 协作平台。同一对话里可能有多个 agent 和用户共处。来自其他 agent 的消息会带 `[@agent_name]` 前缀;你的回复直接说话即可, 不要带前缀。在群聊里可以 @ 别的成员让他们接力 — 仅在你真的需要他们时再 @。 ## 工具与纪律 你能读写文件、改代码、跑命令与测试。 你能用的工具,系统已经注入到这次请求里了 —— **不用自己背 / 列清单**,直接按各工具的 schema 调用。下面只讲规矩: - 写文件**必须**真调 write / edit,落盘成功才算完成;别在没调之前说"已交付 / 已落盘"。 - 报告完成前**必须**调一次 read 把刚写的内容读回来核对。 - 声称"测试通过 / 跑通"前**必须**真用 bash 跑一遍,贴真实输出 + exit_code 为证。 - 给用户汇报讲人话:只说改了哪个文件、干了啥、怎么验证的;别贴 commit hash / git 命令 / 沙箱绝对路径。 ### 几个特殊工具 —— 有就用,没有就忽略 - `ask_user`:需要用户拍板时调它,它会阻塞等回答、同一轮把答案返给你。 - `dispatch`:把活并行派给多个成员。**一次调用**在 `tasks` 里列出全部子任务。正文里 @ 某人只是讨论,**不会真派活、不触发产物合并**。(通常只有协调者有) - `discuss`:让 ≥ 2 名成员就一个话题讨论、**自动收敛**。(通常只有协调者有) ## 你在本项目中的职责 ← ✏️ member_role 用 python 真生成 .docx / .pptx / .xlsx ## 你的人格 / 工作风格 ← ✏️ system_prompt 你是顾屿,后端工程师。 擅长:Python 3.12+ / asyncio / FastAPI / pytest / CLI 工具设计。 工作约束: - 类型注解必须齐全;docstring 在函数体顶 - 单元测试与实现同步 风格:纯函数 + 不可变数据 + 早 return;错误用 raise。 完成报告一句话:"写了 X.py(N 行)+ test_X.py(M 个 case 全通过)"。 不要碰 HTML / CSS / JS / README / CHANGELOG。语气简洁、技术中立。产物注释英文,沟通中文。

<!-- 协调器视角（林知夏拿到） --> # 你是本群聊的协调器(平台职责 — 优先于你的人格设定) 本群由你协调:把用户需求拆成子任务、派给成员、再验收汇总。你自己不写实现代码。 - 派活**只认 `dispatch` 工具**;在正文里 @ 某人、或用 bash / remember 去「宣布」派活,都不算数。 - 子任务需互通时,把规格写进 dispatch 的 `contract`,它会原样发给每个成员。 - 更适合「几个人一起想清楚」时改用 `discuss`。 - **可派活的成员**:顾屿、沈昭、苏念 - 派活后就停,不要轮询;成员完成后你在后续轮验收并向用户汇总。 --- <!-- 普通成员视角（顾屿拿到） --> # 群成员(你可以 @ 谁来讨论) 本群其他成员:@林知夏、@沈昭、@苏念 需要某位队友的意见时,**在回复里 @ ta** 即可把对方拉进来讨论 —— - @ 仅用于**讨论 / 征求意见**,不是派活,也不会触发并行子任务或产物合并。 - 讨论**自动收敛**:别无限互相 @,得出结论就停。

4 Token 预算分配

context/budget.py → compute_budget — 预算 = 模型上下文上限 − 适配器开销（~35k）

层比例最低保底说明
L1 identityavailable × 0.042,000agent 必须知道自己是谁
L4 briefsavailable × 0.063,000项目信息
L5 sharedavailable × 0.083,000契约/决策
L6 activityavailable × 0.1815,000跨对话动态
L7 historyavailable × 0.5435,000最大块 — 历史最重要
L9 user_turnavailable × 0.105,000用户问题

以 claude-sonnet-4（200k 上下文）为例：available = 165k → L1≈6.6k · L4≈9.9k · L5≈13.2k · L6≈29.7k · L7≈89.1k · L9≈16.5k

5 预算执行：2-Pass 截断

context/window.py → enforce_budgets

Pass 1 — 锁定 Hard 层（L1、L9）原样保留，计算剩余预算。

Pass 2 — Soft 层按优先级从高到低分配：超限从顶部（最旧内容）截断，保留标题行；预算耗尽则整层替换为 [此层因总预算耗尽被全部省略]。

截断牺牲顺序（低→高）：L6(p50) → L7(p60) → L5(p55) → L4(p70) → L8(p85) → L2(p95)

6 Token 估算

context/window.py → estimate_tokens — CJK 感知启发式，不依赖 tiktoken：

文本类型CJK 占比公式
CJK 密集>50%chars × 1.5
混合10-50%chars × 1.0
拉丁/代码<10%chars / 3.5

7 隐私隔离

  • R1 项目隔离 — 私聊/非项目对话不注入项目简报（L4）、不注入群成员名单（L2）、活动账本只显示私聊记录
  • R2 角色按项目 — member_role 仅在项目对话注入，不污染全局人格
  • Agent 隔离 — 两个联系人即使共用同一适配器，账本和记忆完全独立，按 agent_id 隔离
  • 可见性 — L6 活动账本只包含 agent 是成员的对话，或同项目下其他对话的动态

10 记忆系统 L5 那一层的内幕

L5「共享记忆」背后是 一张表 conv_memory（与聊天流水 MessageRow、置顶 PinRow 分开）+ 三种 kind，按场景两种读法（ADR-014 / ADR-019）。

kind装什么谁写入
contract共享接口契约（字段/路由/数据形状）dispatch(contract=) 排空时自动种
decision全群拍板的决定remember 工具
artifact谁交付了什么report 收尾自动写

读取按场景分叉 shared.py

判定 workspace_id is None and (direct or members≤2) → 走 B，否则 A：

  • (A) 群聊 / 项目会话 → conv-scoped 共享板：list_conv_memory 取本会话全部，按 kind 分层渲染——契约/决策在前（完整），产物折成 headline。
  • (B) 项目外单聊 → agent-level 记忆：「## 我的工作」(list_agent_memory 跨会话自己记的) + 「## 队友相关工作」(list_workspace_memory 压缩)。配合只读挂载，能真读代码核对。

⚠ list_agent_memory 暂无隐私过滤（纯按 author 跨所有会话捞）→ P1 多用户前要按 workspace 成员资格过滤，同 L6 ledger。

11 编排系统 工具驱动 · boomerang 🪃

没有专用编排代码 —— 靠「一个普通 agent 轮 + 它手里的 dispatch 工具」。入口 dispatch_user_message：群聊必须指定协调者（去中心化模式已移除）；指定者跑 run_adapter_turn(is_dispatcher=True)，非群聊/被@走 1:1。

1 协调者轮跑起来，session 带 role-scoped dispatch 工具
2 它调 dispatch(title, contract, tasks=[{agent,note}…]) 声明并行批次
3 /dispatch 端点 stash 进 _pending_dispatches（不立刻执行）
4 协调者本轮结束 → run_adapter_turn 轮尾 drain：发 BurstCard 泳道 + 把 contract 种进共享记忆 + 为每个 task 并发 spawn 一个 worker（worker prompt = 契约 + 子任务 + 「收尾必须 report + 可 recall」）
5 worker 各自干完翻泳道 done/failed；_mark_burst_task barrier，最后一个完成的认领 last
6 触发：静默 merge to main（auto+workspace）+ 协调者汇总轮（suppress_dispatch，读回各 worker 的 report verdict 核对，从 main 用 present 统一展示交付物面板）

🪃 派活像回旋镖扔出去（fire-and-forget），worker 各自飞回来，最后一个回来的负责收尾 + 叫协调者总结。闭环交接（RuFlo / ADR-014）：worker 必须 report 自评，协调者读回核对，而非「沉默即成功」。partial-success（挂一个不 cancel 全部）+ 1-shot 重试 + _MAX_MENTION_CHAIN_DEPTH 防无限派发。

14 还缺什么

  • Prompt cache 标记：L1/L4 几乎不变但没打 cache_control，每轮重发烧钱。
  • Git commit 集成进 L6：ledger 注释里规划了，代码只查 messages 表，git log 没接。
  • LLM 摘要压缩（P1）：超预算只截断不压缩，老消息直接没了。
  • 真 tokenizer：现用 len(text)//3 估算，中英混合边界会偏。
  • 语义检索：没接 sqlite-vec，远期模糊召回做不到。
  • 隐私过滤补全：list_agent_memory 多用户前需按成员资格过滤。

Polynoia · Design Report · ADR-014

从 RuFlo 学两件事:交接契约 + 共享记忆

RuFlo(ruvnet/ruflo,前身 Claude Flow,40k★)是与我们同赛道的多 Agent 编排框架。 它把 shared memory 放在架构正中心,并把 agent 间的 handoff 做成机制。对照下来,我们有两个真实缺口—— 这版按计划补上这两点,RuFlo 的重型部分(向量库 / federation / 100-agent / 自学习)暂不做。

RuFlo 概念 → 我们现状

同样的四件套,看哪里已对齐、哪里有缺口。

RuFlo含义Polynoia 现状
agents角色限定的 AI 个体✓ 已有 tool_role(orchestrator/coder/designer/writer)
swarms带交互模式的编队🟡 只有「主协调→并行 worker」一种(本轮不扩)

① 交接契约 已采纳

把「所有人必须遵守的接口/字段/路径/端口」从散文提升为结构化字段,逐字下发 + 验收。

1orchestrator 调 dispatch 时填 contract 字段(接口/字段名/路由/端口)——锁一次,不让各人自己发明。
2drain 把 contract 逐字前置到每个 worker 的 prompt:# 接口契约(锁定·不得改动)…
3contract 写进 BurstCard payload → 卡上「契约」可折叠条;刷新仍在。
4收尾汇总轮带上 contract,要求 orchestrator 逐条核对各产物是否符合。

Before

「沈昭的 fetch 路径要和顾屿接口一致」塞在 note 里,靠人眼比对;字段名/端口对不上常要返工。

After

同一份契约下发给三个人,卡上可见,汇总时机器核对。三路产物天然互锁。

改动点:mcp/tools.py::_DispatchTool · routes.py(record_dispatch / drain / 汇总 nudge)· TasksBurstPart.tsx

② 共享记忆 已采纳

会话级、跨 agent 的「已锁定事实」存储,注入每一个 turn。轻量文本,不上向量检索。

1新表 conv_memory(conv / author / kind=contract·decision·artifact / content)。
2任何角色可调 MCP 工具 remember 记一条决策/产物;contract 在 dispatch 时自动入库。
3上下文装配器新增一层 <shared_memory>(L2.5,priority 高于历史)→ 注入每个后续 turn。
4于是 worker 与汇总轮都能看到「全群已锁定的契约/决策/产物」,不再各自重新推导或互相矛盾。

Before

每个 worker 只拿到 conv 历史 + 自己的 note;关键决策散落在长长的聊天里,容易丢。

After

一层精炼的共享事实,稳定在 prompt 顶部;remember 让任何人沉淀「这条大家都得守」。

改动点:storage/models.py(ConvMemoryRow)· storage/repo.py · context/shared.py(新)+ assembler.py · mcp/tools.py(_RememberTool + ROLE_TOOLS)· routes.py(memory 端点 + drain 自动入库)

③ 钩子机制 已研究 · 改用轻量提取(ADR-018)

RuFlo 用 settings.json 声明生命周期事件 + matcher,调起 npx ruflo hooks <name>(~14–17 个 pre/post 钩子)。 我们取思想、不搬机器:RuFlo 是挂在外部 Claude Code 上、要让第三方注册自动化的框架;我们是闭环 app,横切关注点固定。给它装可插拔事件总线 = 过度抽象 + 控制流非局部化。

痛点(workflow 实测)过度方案我们的轻量解
_call 内联六件事(role 门/审计/isError/json/异常)post_tool 事件总线抽 _arg_preview / _result_summary / _wrap_result / _error_result,_call 变薄管线
edit 闸门抄 3 遍(edit/write/apply_patch)pre_edit 钩子 + 注册表一个 _require_edit_approval() helper;lock/commit 因语义不同保留显式
回调 POST「重复」、git_commit message 各异post_tool 回调钩子不动 —— _callback_server 已共享,message 本就该不同(非重复)
turn 路径横切(审计/状态/落盘/编排)pre_turn/on_event/... 钩子层不动 —— is_last 竞态 / 锁 / 重试是临界同步点,包钩子=加风险换化妆

Before · _call

40 行包装器里内联 role 检查 + arg 预览 + tool.start + execute + isError 检测 + tool.end + json.dumps + 异常→tool.error+信封。读一遍才知道发生了啥。

After · _call

一条薄管线:role 门(可否决) → execute() → audit+isError 映射 / on-error 信封。六件横切事各自一个 helper、可单测;加新工具不再抄样板。零行为变化。

何时反悔:若 P1+ 把「用户/插件自定义 per-tool 自动化」列为产品功能,事件总线就从投机抽象变成真需求 —— 届时再建。详见 ADR-018。

改动点:mcp/server.py::_call(+ 模块级 helper)· mcp/tools.py::_require_edit_approval

暂不做 未来可选

RuFlo 的这些是为「企业级 100-agent swarm」准备的,对我们当前阶段过重,先记下。

  • HNSW / AgentDB 向量记忆 + 跨会话语义检索 —— 现在共享记忆是线性文本,够用;量大了再上检索。
  • federation 跨机/跨组织 swarm 协作。
  • swarm 拓扑(mesh / pipeline / adaptive)+ dispatch DAG 依赖(deps/after)—— 即原计划 Phase 3。
  • SONA 自学习 / 学习型路由、多 provider failover。

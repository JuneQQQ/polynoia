2. 组装流程

1 定位 agent — 从 DB 取 Agent 行（名字、人格、后端、模型）
↓
2 逐层构建 — 按 L1 → L1.5 → L2 → L2.5 → L3 → L4 → L4.5 → L5 顺序，每层独立查 DB / 文件系统
↓
3 计算预算 — 根据模型上下文上限减去适配器开销（~35k），按比例分配各层 token 上限
↓
4 预算执行 — 2-pass：先锁定 hard 层不动；soft 层按优先级从高到低分配，超限则从顶部截断
↓
5 拼接输出 — 各层内容用 \n\n---\n\n 连接，形成最终 prompt 字符串

3. 各层详解

L1 · 身份 Identity

源码 context/identity.py → build_identity_layer

优先级 100 · HARD — 永不截断

告诉 agent "你是谁"，包含：

  • 名字、handle、id、后端适配器、模型
  • 平台简介（Polynoia 是什么、MCP 工具体系）
  • 工具纪律 — 平台按 tool_role 注入「能力边界 + 使用规矩」，行为化、不枚举工具名：工具 schema 已由 MCP 发现并注入到请求的 tools 字段，模型本就看得见有哪些工具，再在 prompt 里列一遍是冗余、还会和实际注入集漂移。ask_user / dispatch / discuss 等特殊工具以「有就用，没有就忽略」的条件写法并入此处
  • 项目内职责 — 仅群聊项目对话才注入 member_role，私聊不出现
  • 人格/工作风格 — 用户在"新建联系人"里写的 system_prompt（纯人格：身份/领域/语气/产物语言；工具与纪律一律交给平台注入，persona 里不再写）

设计变更（2026-06-02）：人格 prompt 不再自带工具枚举或「工具使用纪律」块；这些统一由本层（identity.py）平台注入，单一来源，避免和 MCP 注入的 tools 字段重复/打架。

# 身份 你是 **顾屿**(handle:`@顾屿`,id:`01JX...`)。 由 `claudeCode` 后端驱动,model = `claude-sonnet-4-6`。 ## 关于平台 你运行在 Polynoia 平台上。Polynoia 是一个多 agent 协作平台…… ## 工具与纪律 你能读写文件、改代码、跑命令。 ← 行为化能力边界(按 tool_role,不列工具名) 你能用的工具系统已注入这次请求,不用自己背/列清单——按 schema 调用即可。下面只讲规矩: - 写文件必须真调 write/edit,落盘才算完成;报告前 read 回核;声称"测试通过"前真跑 bash 贴 exit_code。 - 给用户讲人话:只说改了哪个文件/干了啥/怎么验证;别贴 commit hash / git 命令 / 沙箱路径。 ### 几个特殊工具——有就用,没有就忽略 - `ask_user`:需要用户拍板时调它,它会阻塞等回答、同一轮把答案返给你(别瞎猜、别写"等用户指令")。 - `dispatch`:并行派活,一次调用在 tasks 里列全;@提及只是讨论,不算派活。(通常只有协调者有) - `discuss`:让≥2名成员就一个话题讨论并自动收敛(权衡/评审/共识)。(通常只有协调者有) ## 你在本项目中的职责 后端开发，负责 API 和数据库层 ## 你的人格 / 工作风格 严谨、注重代码质量……

L1.5 · 协调器协议 / 群成员名单

源码 context/orchestrator.py + context/group_members.py

条件 仅群聊（conv.group = True）注入，私聊不出现 R1

  • 协调器（orchestrator_member_id == agent_id）→ 注入强制派活指令 + 可派活成员名单，优先级 99，HARD。ADR-017 这层只负责「兜底强制」(就算人格只字不提派活，协调器也一定拿到「派活只认 dispatch」)；dispatch/discuss 的具体用法已并入 L1 的工具规则，这里不再重复一遍
  • 普通成员 → 注入"你可以 @ 谁来讨论"名单 + 讨论收敛提示，优先级 95，soft

设计变更（2026-06-02）：本层从原来重复的 dispatch/discuss how-to 大块，瘦成只留承重部分（强制「派活只认 dispatch」、@提及≠派活、可派成员名单）。用法指向 L1 工具规则，单一来源。

# 你是本群聊的协调器(平台职责 — 优先于你的人格设定) 本群由你协调:拆需求 → 派活 → 验收 → 汇总;你自己不写实现代码。 - 派活只认 `dispatch` 工具(用法见上面的工具规则);@提及 / bash「宣布」都不算数。 - 需要权衡/评审/共识(而非拆独立产物)时改用 `discuss`。 - 可派活的成员:顾屿、沈昭 - 派活后就停,不要轮询;成员完成后你在后续轮验收并汇总。 --- # 群成员(你可以 @ 谁来讨论) 本群其他成员:@顾屿、@沈昭 需要某位队友的意见时,**在回复里 @ ta** 即可把对方拉进来讨论……

L2 · 项目简报 Project Briefs

源码 context/briefs.py → build_project_briefs_layer

条件 仅项目对话注入；私聊/非项目对话返回 None R1

当前对话所在项目的名称、仓库、描述、成员 + 你也是哪些项目的成员（最多 10 个）。

# 你的项目 ## 当前对话所在项目:**后端重构** · 仓库 `github.com/.../backend` 从单体拆成微服务 成员:@顾屿, @沈昭 ## 你也是这些项目的成员 - **前端重写** · `github.com/.../frontend` - **数据分析** · `github.com/.../analytics`

L2.5 · 共享记忆 Shared Memory

源码 context/shared.py → build_shared_memory_layer

两种模式：

  • 群聊/项目对话 → 该对话的共享黑板（contract / decision / artifact），由 remember 工具和 dispatch 自动写入 ADR-014
  • 项目外私聊 → agent 自己的工作记忆（跨对话持久） ADR-019
<shared_memory> # 共享记忆(本群已锁定的契约 / 决策 / 产物 — 必须遵守) ## 契约 - API 返回格式统一为 {code, data, message} ## 决策 - [顾屿] 使用 PostgreSQL 而非 SQLite ## 产物 - [沈昭 自评:ok · 契约符合] migration.py — 数据库迁移脚本 </shared_memory>

L3 · 活动账本 Activity Ledger

源码 context/ledger.py → build_activity_ledger_layer

跨对话的最近动态，让 agent 知道"其他地方在发生什么"：

  • 可见性：agent 是成员的对话 + 同项目下其他对话的动态
  • 排除当前对话（L4 已覆盖）
  • 内容：最近消息摘要 + git 提交记录
  • 按项目分组，私聊单独一节
  • 每对话最多 8 条，总计最多 30 条
# 你的近期活动(其他对话) ## 项目 · 后端重构 ### API 设计讨论 (群聊) - _5分钟前_ @沈昭: 返回格式统一为 JSON - 📝 _代码变更_: · `a3f2c1b` refactor: unify API response format ## 私聊 ### 和用户的对话 (DM) - _1小时前_ 用户: 帮我看看这个报错

L4 · 对话历史 Conv History

源码 context/history.py → build_conv_history_layer

当前对话的滚动窗口（默认最近 30 条，oldest→newest）：

  • 每条消息标注发送者：[user] / [@顾屿] / [@顾屿(你)]
  • 单条消息上限 8k token，超限则头尾保留 + 折叠标记
# 本对话历史(最近 N 条) [user]: 帮我重构一下用户模块 [@顾屿(你)]: 好的，我先看看现有代码结构…… [@沈昭]: 我觉得应该先定接口契约 [user]: 同意，先定契约再动手

L4.5 · 置顶消息 Pinned

源码 context/assembler.py 内联构建

用户手动 pin 的关键消息，优先级 85（高于历史），作为长期上下文注入。

L5 · 用户当前消息 User Turn

优先级 90 · HARD — 永不截断

触发本轮回复的用户原文。即使用户贴了 20k token 的代码，也不截断——截断用户的问题 = 答案一定错。其他层会先被挤掉。

# 当前用户消息 帮我把这个函数改成异步的，这是现在的代码： ```python def get_user(id): ... ```

4. Token 预算分配

源码 context/budget.py → compute_budget

预算 = 模型上下文上限 − 适配器开销（Claude Code ~35k）。各层按比例分配：

层比例最低保底说明
L1 identityavailable × 0.042,000agent 必须知道自己是谁
L2 briefsavailable × 0.063,000项目信息
L2.5 sharedavailable × 0.083,000契约/决策（从 history 份额切出）
L3 activityavailable × 0.1815,000跨对话动态
L4 historyavailable × 0.5435,000最大块 — 历史最重要
L5 user_turnavailable × 0.105,000用户问题

以 claude-sonnet-4（200k 上下文）为例：

  • available = 200k − 35k = 165k
  • L1 ≈ 6.6k · L2 ≈ 9.9k · L2.5 ≈ 13.2k · L3 ≈ 29.7k · L4 ≈ 89.1k · L5 ≈ 16.5k

5. 预算执行：2-Pass 截断

源码 context/window.py → enforce_budgets

Pass 1 — 锁定 Hard 层

L1（身份）和 L5（用户消息）原样保留，不截断。计算 hard 层已用 token，剩余 = 全局预算 − hard 已用。

Pass 2 — Soft 层按优先级分配

Soft 层按优先级从高到低排序，依次分配：

  • 每层取 min(该层上限, 剩余全局预算) 作为有效上限
  • 超限 → 从顶部（最旧内容）截断，保留标题行
  • 预算耗尽 → 整层替换为 [此层因总预算耗尽被全部省略]

截断优先级排序（低→高，先被牺牲的）：

  • L3 activity (p50) — 最先被砍
  • L4 history (p60)
  • L2.5 shared (p55)
  • L2 briefs (p70)
  • L4.5 pinned (p85)
  • L1.5 members (p95)

6. Token 估算

源码 context/window.py → estimate_tokens

不依赖 tiktoken，用 CJK 感知启发式：

文本类型CJK 占比公式
CJK 密集>50%chars × 1.5
混合10-50%chars × 1.0
拉丁/代码<10%chars / 3.5

原因：chars / 3 对中文严重低估（1 汉字 ≈ 1.5-2 token 但只算 1 char）。

7. 隐私隔离

  • R1 项目隔离 — 私聊/非项目对话不注入项目简报（L2）、不注入群成员名单（L1.5）、活动账本只显示私聊记录
  • R2 角色按项目 — member_role 仅在项目对话注入，不污染全局人格
  • Agent 隔离 — 两个联系人即使共用同一适配器（如两个 Claude 联系人），账本和记忆完全独立，按 agent_id 而非 adapter_id 隔离
  • 可见性 — L3 活动账本只包含 agent 是成员的对话，或同项目下其他对话的动态

字段编辑位置注入到场景

## 0. 环境准备

- 起服务:`make dev`(前端 :7788 + 后端 :7780)。
- 健康检查:
  - 前端:`http://127.0.0.1:7788`
  - 后端:`http://127.0.0.1:7780/api/health`(`/health` 不是后端健康检查入口)。
- 清库回初始态(测试前常用):
  - `python3 scripts/reset_clean.py --keep-contacts` —— 留联系人/适配器,清会话/工作区/消息 + 沙箱。
  - `python3 scripts/reset_clean.py`(全清,需输 `yes`)—— 回到空白基线,重新建联系人。
  - 清完**重启 server**(内存中的 adapter session / burst 状态指向已删数据)。
- 改了前端代码:浏览器**硬刷新**;改了依赖/后端:**完整重启 `make dev`**。

## 1. 冲突闭环(conflict closed-loop) —— 最早的核心案例

前置:一个**项目群聊**(有 workspace),里面有协调器 + ≥2 个写手联系人(见 §2 协调器)。

### TC1.1 制造冲突 + 浮现冲突卡
- 操作:给群聊发任务,让协调器**派两个写手改同一个文件的同一处**(例如都改 `app.py` 第 N 行 / 都新建同名 `VERSION.txt` 写不同内容)。
- ✅ 预期:两个写手各自在自己分支写完 → 合并 main 时第二个**失败** → 时间线出现**琥珀色冲突卡**(`待解决`,列出冲突文件 + 类型如「双新增/内容/改删/二进制」)。
- ⚠️ 注意:必须走**调度 dispatch → burst → 合并**才会触发(纯 @ 提及不触发合并阶段)。跨会话共享 workspace 残留会导致 main 已有内容、分支 ahead=0 不冲突 → 先 `reset_clean.py` + 新建群聊。

### TC1.2 解决冲突(人工)
- 操作:点冲突卡「解决冲突」→ 右栏出解决面板。
- ✅ 预期:
  - 顶部大白话说明:`{码乙} 的改动和 {码甲} 已合入 main 的版本 改到了同一处`(来源标注,不是抽象 main)。
  - 每个文件一个**折叠 diff**(长文件未变行折叠,只显示变化处)+「N 处差异」徽标 + 列图例(主线侧 / 某 agent 版本)。
  - 按钮:`采用 {码甲} / 采用 {码乙} / 手动合并`(modify_delete 是 保留/删除;binary 是整体采用一侧)。
- 操作:选一侧(或手动合并编辑)→「解决并合并」。
- ✅ 预期:卡翻绿色「已解决 → main@<sha>」;右栏代码区**自动刷新**(文件落 main)。

### TC1.3 放弃冲突
- 操作:冲突卡点「放弃」。
- ✅ 预期:卡翻「已放弃,分支未合并进 main」;不再重复浮现(dedup)。
- 并发安全:放弃与解决并发时,已合并进 main 的不会被错标 abandoned(加了 workspace 锁)。

## 2. 群聊协调器「自带电」(ADR-017)

- 操作:自建一个普通联系人(默认 `generalist`,无 dispatch 工具),把它指为某群聊的协调器(`orchestrator_member_id`),发个需要拆分的任务。
- ✅ 预期:它**能 dispatch 起 burst**(不依赖 persona 写没写派活指令),且物理上**没有 write 工具**(只调度不自己写)。同一联系人在别的群可当写手,互不串。

## 3. 文档 / PPT / 表格 预览 + 导出

准备(让 agent 产出测试文件,发群聊):
```
帮我在项目里建三个文件:
1. docs/intro.md —— 中文 Markdown,用上标题/加粗/列表/代码块/表格/引用/链接,约 300 字。
2. docs/slides.md —— Marp 幻灯,开头第一行起必须是:
   ---
   marp: true
   theme: default
   paginate: true
   ---
   再用单独一行 --- 分 6 页(封面/背景/功能/架构/进展/谢谢)。
3. data/sample.csv —— 表头「产品,单价,月销量,地区」+ 8 行中文数据。
都写进项目工作区。
```

### TC3.1 文件自动刷新
- ✅ agent 写完 + 合并进 main 后,**不手动刷新**,右栏代码区自动出现 `docs/`、`data/` 及文件,并自动打开一个。

### TC3.2 文档 WYSIWYG 编辑
- 打开 `docs/intro.md` → 预览 → ✅ Word 式排版文档(标题/列表/表格/代码块都渲染)、内容居中有页边距、可直接点字编辑。
- 改几个字 → ✅ 顶部出现「未保存」点 → Ctrl+S → 变「已保存」;切代码 tab 源码已同步。

### TC3.3 文档导出
- intro.md 预览右上角:✅ `PDF`(打印对话框→另存 PDF)/ ✅ `.md`(原文)/ ✅ `.docx`(Word 能打开,排版基本还原)。

### TC3.4 PPT 渲染 + 导出
- 打开 `docs/slides.md` → 预览 → ✅ 翻页幻灯(6 页 + 页码);代码 tab 改源码 → 预览实时变。
- ✅ `PDF`(每页一张幻灯)/ ✅ `.pptx`(PowerPoint 能打开,纯文本标题+正文,**不带 Marp 主题样式**——预期,要样式用 PDF)。

### TC3.5 表格展示 + 导出
- 打开 `data/sample.csv` → 预览 → ✅ Excel 式表格(灰底表头 + 网格)。
- ✅ `.xlsx`(Excel 能打开)/ ✅ `.csv` / ✅ `PDF`。

### TC3.6 静态 HTML 预览
- 让 agent 写一个 `index.html`(内联 CSS/JS 的简单页面)→ 打开 → 预览 → ✅ iframe 渲染出页面;✅ 可导出 `PDF` / `.html` 原文。

### TC3.7 不支持类型兜底
- 打开 `.py` → 预览 → ✅ 提示「不是文档/幻灯/网页,用代码区编辑」(不白屏不报错)。

> 导出边界(非 bug):PDF 走浏览器打印对话框;`.docx` 排版打折;`.pptx` 纯文本无主题。更高保真的 .docx/.pptx 属后端方案(后续)。

## 4. IM / 群聊按钮矩阵

前置:一个项目群聊,包含 `you` + 协调者 + 至少 3 个联系人。建议角色覆盖:协调者、写作型、脚本型、数据型。

### TC4.1 会话列表行按钮
- 操作:会话行点击正文区域。
- ✅ 预期:进入该会话;标题、成员数、头像堆叠与左侧列表一致。
- 操作:会话行 `⋯`。
- ✅ 预期:菜单包含 `编辑成员`、`群聊设置`、`归档群聊`;不应出现 `删除对话`(未归档群聊不能直接删)。
- 操作:`编辑成员` → 添加一个项目成员 → 保存。
- ✅ 预期:群聊成员数 +1;标题区和侧边栏同步;聊天流出现 `成员变更` 系统事件;新成员进入后能看到该事件作为上下文边界。
- 操作:再次 `编辑成员` → 移除刚加入的成员 → 保存。
- ✅ 预期:成员数 -1;聊天流出现 `成员变更` 系统事件;被移除成员不再被 @ 候选命中。
- 操作:`群聊设置` → 改某成员本对话职责 → 保存。
- ✅ 预期:聊天流出现 `角色更新` 系统事件;成员详情页显示新职责;后续上下文应包含“角色更新”事件,避免 agent 只依赖旧角色。
- 操作:`归档群聊`。
- ✅ 预期:出现二次确认弹窗;确认后会话从普通列表消失,进入归档列表。

### TC4.2 标题区按钮
- 操作:标题区成员数。
- ✅ 预期:右侧成员抽屉打开;成员列表只显示当前群聊成员,不是项目全员。
- 操作:成员抽屉 `添加成员`。
- ✅ 预期:候选只来自项目成员且排除已在群聊的人;添加后列表、标题、侧栏同步刷新。
- 操作:标题区头像。
- ✅ 预期:打开对应 Agent 详情。普通成员详情只保留 `移除群聊`;协调者不显示 `移除群聊`。
- 操作:右栏关闭按钮。
- ✅ 预期:右栏关闭后主聊天布局恢复,输入框不漂移。

### TC4.3 @ 与群聊调度
- 操作:输入框键入 `@`。
- ✅ 预期:@ 候选浮层出现;候选只包含当前群聊成员,支持键盘/点击选择。
- 操作:发送 `@某成员 smoke`。
- ✅ 预期:消息中渲染 mention chip;点击 chip 打开该成员详情;若触发 agent 执行,状态条出现正在执行状态。
- 操作:运行时点击状态条或中断按钮。
- ✅ 预期:执行被中断;聊天流保留错误/中断块,刷新页面后错误块仍在,不能丢失或被“清空”。
- 操作:让协调者派多个子任务。
- ✅ 预期:只渲染真实运行的子 agent 泳道;不存在的分工不应占空泳道;派活卡显示 running/done/failed 状态。

### TC4.4 输入框按钮
- 操作:空输入点击发送。
- ✅ 预期:不发送空消息,不触发 agent。
- 操作:输入文字后发送。
- ✅ 预期:按钮变为可用;消息入流;输入框清空;不会重复发送。
- 操作:点击附件按钮。
- ✅ 预期:文件选择器打开;取消后输入框状态不丢;大文件超过限制时显示明确错误。
- 操作:拖文件到输入框。
- ✅ 预期:输入框高亮;上传中有状态;刷新/关闭页面时有上传警告;上传完成后附件可发送。
- 操作:输入框聚焦、换行、多行文本。
- ✅ 预期:输入框自动增高但不遮挡最近消息;执行状态面板和输入框视觉连在一起。

## 5. 单聊 / main 同步 / 文件按钮

前置:一个项目单聊,成员为 `you` + 一个 coding agent。

### TC5.1 单聊基础按钮
- 操作:会话列表点击单聊。
- ✅ 预期:进入单聊;输入框 placeholder 显示目标 agent 名;不显示群聊 @ 提示。
- 操作:联系人列表点击某 agent。
- ✅ 预期:进入或创建 1v1 单聊;若已有会话则复用,不重复建同名单聊。
- 操作:空发送、附件、输入发送。
- ✅ 预期:行为同 TC4.4;单聊不显示 Auto/Manual 分段控件。

### TC5.2 agent 产出同步 main
- 操作:让单聊 agent 新建一个小文件,例如 `qa-single-main-sync.md`。
- ✅ 预期:agent 结束后文件自动合入 workspace main;右侧文件树自动刷新;无需手动点刷新。
- 操作:打开文件预览 → 代码 tab 改内容 → 保存。
- ✅ 预期:保存直接写 main 并生成 `用户编辑 ...` commit;提交历史页能看到该 commit。
- 操作:刷新页面。
- ✅ 预期:文件仍在;提交历史仍能查到;不存在只写在 agent worktree 的“幽灵文件”。

### TC5.3 查看完整文件与撤销
- 操作:打开已合并 diff 卡,点击 `撤销`。
- ✅ 预期:在 workspace main 上反向应用 diff,并新增 revert commit;文件树同步刷新。
- 操作:撤销一个已经被后续修改覆盖的 diff。
- ✅ 预期:失败时显示“文件状态已变化/无法干净撤销”的明确错误,不能静默成功。
- 操作:查看完整文件。
- ✅ 预期:若保留该入口,默认打开当前 main 的完整文件;若 diff 未合并才 fallback 到 agent worktree;如果实现不了该语义,按钮应隐藏。

- 操作:失败工具调用块刷新页面。
- ✅ 预期:错误块原样保留,包括输入、输出错误、状态;不能刷新后消失。

- 操作:打开右侧代码区 → 提交历史。
- ✅ 预期:默认显示 main 提交列表,最新提交在前;每条有标题、sha、时间、作者。

- ✅ 预期:树模式显示提交泳道;列表模式恢复普通列表;切换不丢选中提交。

- 操作:折叠文件 diff。
- ✅ 预期:该文件内容收起,标题和统计保留;再次点击恢复。
- 操作:点击 `撤销`。
- ✅ 预期:撤销在 main 上产生 revert commit;撤销按钮状态变为已撤销或给出失败原因。

- 回滚原因:对"实时渲染"太重(起容器 + 装依赖几十秒)、大项目环境千奇百怪难适配、达不到实时。改为**文档/PPT/表格前端渲染**(§3),大项目交互改走**后续的右侧终端**(见 `docs/design/right-rail-roadmap.md`)。
- 存档案例(如将来重启该方向可复用):
  - static:workspace 有 `index.html` → 起 `python -m http.server` → curl 到页面。
  - python:`app.py`(Flask)无 requirements → import 推断装 flask/flask_cors → sitecustomize 强制 `0.0.0.0:8000` → 访问首页。
  - 状态机:starting(HTTP 探测未通)→ running(探测通)→ error(容器退出,附 logs)。
